# AI Agent 轻量级协同系统设计文档 v2.0

## 1. 文档目标

本文档定义一个**可扩展的通用多 Agent 协同框架首版设计**，用于支持“围绕单一复杂任务，由多个不同角色 Agent 分工协作完成”的业务场景。

本版本重点解决以下问题：

1. 保留通用多 Agent 框架的扩展能力
2. 避免首版过度设计
3. 明确状态真源、控制范式与失败语义
4. 便于 AI coder 按阶段开发、自测与验收

本文档替代上一版中边界不清、首版范围过宽的部分，作为后续编码与测试的统一依据。

---

## 2. 设计原则

### 2.1 首版目标

首版系统只解决以下问题：

- 一个复杂任务的拆解、执行、审核与记录闭环
- 多角色 Agent 的协作与结果留痕
- 任务状态可恢复、过程可审计、结果可复盘
- 代码可由 AI coder 按阶段实现并通过自动化测试验证

### 2.2 非目标

首版**不追求**以下能力：

- 分布式多机部署
- 高并发任务池调度
- 强一致消息队列
- 长驻守护型自治 Agent 集群
- 自动上下文压缩驱动的复杂记忆系统
- 数据库后端或远程存储

### 2.3 核心原则

1. **半中心化控制**：由 Orchestrator 推进流程，Agent 不直接自治调度全局任务。
2. **文件为状态真源**：所有关键状态、任务、产物、日志必须落盘，内存仅作为执行缓存。
3. **首版单次调用**：Agent 默认采用“一次调用型”，接到任务后执行一次并返回结果。
4. **审计优先于性能**：优先保证流程可观察、可恢复、可测试。
5. **先闭环，再扩展**：首版先保证 Planner → Executor → Reviewer → Record 闭环跑通，再逐步扩展自治能力。

## 3. 总体架构

### 3.1 控制范式

本系统首版采用**半中心化控制架构**。

职责划分如下：

- **Orchestrator**：唯一流程控制中枢，负责推进工作流、调用 Agent、处理失败与重试。
- **TaskManager**：负责任务状态、依赖关系、任务选择与状态变更。
- **AgentRunner**：根据角色执行当前任务，生成结构化结果。
- **ToolSandbox**：提供工具注册、参数校验、执行保护与错误标准化。
- **WorkspaceStore**：负责文件系统读写、目录初始化、原子写入。
- **AuditLog**：统一记录关键过程事件。

### 3.2 首版角色

首版定义四类角色：

- **Planner**：将复杂任务拆解为可执行子任务，并给出输入输出与验收要求。
- **Executor**：执行具体子任务，生成产物。
- **Reviewer**：审核执行结果，决定通过、驳回或补充修改要求。
- **Recorder**：在首版中不作为独立常驻 Agent 存在，由 Orchestrator 统一完成关键事件记录。

说明：

- “记录”能力在 v2 中保留为系统能力，而非独立调度角色。
- 如果后续业务需要独立总结/沉淀 Agent，可在 v3 中将 Recorder 独立出来。

## 4. 工作空间与持久化结构

所有协同数据必须落盘至单个任务工作空间目录。

### 4.1 目录结构

```text
/workspaces/task_{uuid}/
 ├── meta.json
 ├── state.json
 ├── context.json
 ├── event_log.jsonl
 ├── tasks/
 │    ├── task_root.json
 │    ├── task_plan_1.json
 │    ├── task_exec_1.json
 │    └── task_review_1.json
 └── artifacts/
      ├── draft_1.md
      ├── review_1.json
      └── final_output.md
```

### 4.2 文件说明

#### `meta.json`
只读元数据，记录任务初始输入与全局约束。

示例字段：

```json
{
  "task_id": "task_xxx",
  "title": "string",
  "created_at": "iso8601",
  "input": {
    "user_request": "string"
  },
  "config": {
    "retry_limit": 2,
    "max_review_round": 2
  }
}
```

#### `state.json`
当前工作流全局状态。

```json
{
  "workflow_status": "planning | executing | reviewing | completed | failed",
  "current_task_id": "string | null",
  "current_role": "planner | executor | reviewer | null",
  "retry_limit": 2,
  "review_round": 0,
  "last_error": null
}
```

#### `context.json`
共享上下文摘要，用于跨阶段传递必要的结构化信息。

说明：

- `context.json` 只保存提炼后的关键上下文
- 不能替代 task、artifact、event_log
- 事实真源仍然是任务文件与产物文件

#### `event_log.jsonl`
统一审计流。每一行一个标准化事件。

#### `tasks/*.json`
任务单集合。

#### `artifacts/*`
执行产物、审核结果、最终输出等文件。

## 5. 核心数据模型

### 5.1 Task Node Schema

首版任务单必须采用可执行任务模型，而不是仅表达待办事项。

```json
{
  "id": "string",
  "subject": "string",
  "description": "string",
  "role": "planner | executor | reviewer",
  "status": "pending | in_progress | completed | failed",
  "owner": "string | null",
  "blockedBy": ["string"],
  "input_refs": ["string"],
  "output_refs": ["string"],
  "parent_task_id": "string | null",
  "retry_count": 0,
  "failure_reason": "string | null",
  "priority": 100,
  "created_at": "iso8601",
  "updated_at": "iso8601"
}
```

### 5.2 字段说明

- `role`：该任务预期由哪类 Agent 执行
- `blockedBy`：依赖的任务 ID 列表
- `input_refs`：输入文件引用
- `output_refs`：输出文件引用
- `parent_task_id`：父任务 ID，用于表达任务层级
- `retry_count`：重试次数
- `failure_reason`：失败原因，失败后必须填写
- `priority`：优先级，数值越小优先级越高

### 5.3 Event Schema

首版统一事件模型如下：

```json
{
  "event_id": "uuid",
  "timestamp": "iso8601",
  "kind": "workflow | task | agent | tool",
  "type": "string",
  "task_id": "string | null",
  "role": "string | null",
  "status": "string | null",
  "summary": "string",
  "details": {
    "artifact_refs": [],
    "error": null,
    "extra": {}
  }
}
```

说明：

- 首版保留统一事件流，不强制引入强一致 inbox 消息总线
- 如后续需要异步 Agent 通知，可在此基础上扩展 MessageBus

### 5.4 Tool Result Schema

所有工具执行结果必须标准化返回：

```json
{
  "status": "ok | failed",
  "output": "string | object | null",
  "error": "string | null"
}
```

## 6. 模块设计

### 6.1 WorkspaceStore

#### 功能

- 初始化工作空间目录
- 原子写入 JSON / JSONL / 文本文件
- 读取状态文件、任务文件、产物文件

#### 约束

- 对 `state.json`、`tasks/*.json` 等覆盖写，必须采用写时复制
- 禁止直接覆盖写破坏文件完整性

#### 原子写要求

流程必须为：

1. 写入 `xxx.tmp`
2. `flush + fsync`
3. `os.replace(tmp, target)`

#### 不负责

- 不负责流程控制
- 不负责任务调度
- 不负责模型调用

---

### 6.2 TaskManager

#### 功能

- 创建任务
- 获取当前可执行任务
- 更新任务状态
- 解除任务依赖
- 记录失败原因与产物引用

#### 核心接口

```python
create_task(task: dict) -> None
get_next_runnable_task() -> dict | None
update_task(task_id: str, patch: dict) -> None
mark_task_completed(task_id: str, output_refs: list[str]) -> None
mark_task_failed(task_id: str, reason: str) -> None
```

#### 任务选择规则

可执行任务必须同时满足：

- `status == pending`
- `blockedBy == []`

首版按以下顺序选取：

1. `priority` 最小
2. `created_at` 最早

#### 边界说明

首版 TaskManager 不解决多进程并发 claim 问题。
它假设只有一个 Orchestrator 进程负责推进工作流。

---

### 6.3 Orchestrator

#### 功能

- 驱动主工作流
- 按任务角色调用对应 AgentRunner
- 处理任务成功、驳回、失败与重试
- 写入 `state.json`
- 写入 `event_log.jsonl`

#### 控制职责

Orchestrator 是首版唯一的流程推进者，必须负责：

1. 初始化根任务
2. 调用 Planner 进行拆解
3. 为每个执行任务调用 Executor
4. 为执行结果调用 Reviewer
5. 根据 Reviewer 结论决定：
   - 通过并结束
   - 重派 Executor
   - 达到上限后失败结束

#### 核心接口

```python
run_workflow(task_id: str) -> None
invoke_role(role: str, task: dict) -> dict
handle_review_result(review_result: dict, task: dict) -> None
append_event(event: dict) -> None
```

---

### 6.4 AgentRunner

#### 功能

- 根据 `role` 组装 prompt
- 读取任务输入与必要上下文
- 调用 Mock LLM 或真实 LLM
- 返回结构化结果

#### 首版执行模式

首版只支持**同步单次调用**。

执行流程：

1. 读取当前任务及 `input_refs`
2. 读取必要 `context.json`
3. 组装角色 prompt
4. 调用模型
5. 输出标准化结果
6. 返回给 Orchestrator

#### 不做

- 不做 idle loop
- 不做长期挂起
- 不做常驻轮询
- 不做自动身份重注

#### 返回结果建议

```json
{
  "status": "ok | failed",
  "summary": "string",
  "artifact_refs": ["string"],
  "next_action": "complete | retry | reject",
  "structured_output": {}
}
```

---

### 6.5 ToolSandbox

#### 功能

- 工具注册
- 参数校验
- 安全执行
- 异常捕获
- 返回标准化结果

#### 接口形式

```python
@tool(name, desc, schema)
def some_tool(...):
    pass

execute_tool(tool_name: str, kwargs: dict) -> dict
```

#### 错误处理要求

所有工具执行必须包裹在 `try-except` 中。
任何异常都不能直接导致 AgentRunner 崩溃，必须转化为标准结果：

```json
{
  "status": "failed",
  "output": null,
  "error": "KeyError: url"
}
```

---

### 6.6 AuditLog

#### 功能

- 将所有关键事件统一记录到 `event_log.jsonl`
- 为调试、复盘、恢复提供依据

#### 必须记录的事件

- workflow 启动
- task 创建
- task 开始执行
- task 完成
- task 失败
- reviewer 驳回
- reviewer 通过
- 工具失败
- workflow 终止

## 7. 工作流设计

### 7.1 标准闭环

首版标准流程如下：

1. 用户输入复杂任务
2. Orchestrator 创建根任务
3. Planner 拆解子任务并定义验收标准
4. Executor 执行子任务并生成产物
5. Reviewer 审核产物
6. Orchestrator 根据审核结果推进：
   - pass：标记完成
   - reject：重派 Executor
   - 超过重试上限：整体失败
7. 输出最终结果与审计日志

### 7.2 时序示意

```text
User -> Orchestrator: submit task
Orchestrator -> Planner: invoke planning
Planner -> Orchestrator: sub tasks + acceptance
Orchestrator -> Executor: invoke execution
Executor -> Orchestrator: artifact refs + summary
Orchestrator -> Reviewer: invoke review
Reviewer -> Orchestrator: pass / reject
Orchestrator -> WorkspaceStore: update task/state/log
Orchestrator -> User: final output
```

## 8. 失败模型与恢复语义

### 8.1 失败分类

系统统一定义四类失败：

#### 1. tool_failed
工具调用失败。

特点：
- 发生在单步执行内部
- 可以在当前任务上下文内重试
- 必须记录工具错误

#### 2. task_failed
单个任务未完成或结果不符合要求。

特点：
- 任务级失败
- 必须写入 `task.failure_reason`
- 可由 Orchestrator 决定是否重试

#### 3. workflow_failed
关键路径无法继续推进，整个工作流失败。

特点：
- 全局失败
- 必须写入 `state.json.last_error`
- 必须写入终止事件

#### 4. aborted
人工或系统主动中止。

### 8.2 重试规则

- `tool_failed`：允许任务内局部重试
- `task_failed`：允许按 `retry_limit` 重派
- `workflow_failed`：终止，不自动恢复

### 8.3 记录规则

失败后必须同时：

1. 更新任务状态
2. 填写失败原因
3. 写入 event_log
4. 不覆盖旧记录，只追加新事件

## 9. 并发与一致性边界

### 9.1 首版边界

本系统首版明确只支持：

- 单个 Orchestrator 进程
- 单工作空间串行推进
- 轻量任务流

### 9.2 不保证的能力

首版**不保证**：

- 多进程并发领取同一任务的强一致性
- 多消费者并发消费同一消息队列
- 跨文件事务一致性

### 9.3 说明

文件原子替换只保证单文件写入完整性，不保证多文件联合提交。
因此首版通过“单 Orchestrator 推进”规避复杂并发一致性问题。

## 10. 可扩展点设计

虽然首版收紧实现范围，但必须保留扩展接口。

### 10.1 StorageBackend

当前默认实现：文件系统
后续可扩展：SQLite、本地 KV、远程对象存储

建议接口：

```python
class StorageBackend:
    def read_json(self, path): ...
    def write_json_atomic(self, path, data): ...
    def append_jsonl(self, path, event): ...
    def read_text(self, path): ...
    def write_text(self, path, text): ...
```

### 10.2 AgentRunner Backend

当前默认实现：Mock LLM / 本地模型调用
后续可扩展：OpenAI、Anthropic、其他模型提供方

### 10.3 ToolRegistry

当前默认实现：本地 Python 函数
后续可扩展：远程工具服务、MCP 风格工具代理

### 10.4 Lifecycle Mode

当前默认实现：single-shot
后续可扩展：daemon / polling / event-driven

## 11. TDD 与 AI Coder 开发契约

本章节是给 AI coder 的**硬性开发约束**，不是建议项。

### 11.1 总体要求

- 必须采用分阶段开发
- 必须先写测试，再写实现
- 每阶段测试未通过，不得进入下一阶段
- 不得一次性生成整套系统代码

### 11.2 阶段 1：Workspace 与 Schema

#### 允许实现

- 工作空间初始化
- 原子写入
- Task/Event/State Schema 校验

#### 禁止实现

- AgentRunner
- Orchestrator 主循环
- 真实模型调用

#### 必须通过的测试

- `test_workspace_init`
- `test_atomic_write`
- `test_task_schema_validation`
- `test_event_append`

### 11.3 阶段 2：TaskManager

#### 允许实现

- 创建任务
- 任务选择
- 状态更新
- 依赖解除
- 失败原因记录

#### 禁止实现

- 多 Agent 常驻循环
- 真实消息队列

#### 必须通过的测试

- `test_create_task`
- `test_get_next_runnable_task`
- `test_unblock_after_complete`
- `test_mark_task_failed`

### 11.4 阶段 3：Orchestrator + Mock Agent

#### 允许实现

- 单次角色调用
- Mock LLM
- 产物落盘
- 审计日志记录
- 重试控制

#### 禁止实现

- 真实外部模型 API
- 自动上下文压缩
- 常驻 Agent 模式

#### 必须通过的测试

- `test_planner_executor_reviewer_flow`
- `test_review_reject_and_retry`
- `test_workflow_fail_after_retry_limit`
- `test_event_log_complete_trace`

### 11.5 阶段 4：真实模型接入（可选）

仅在前三阶段全绿后允许进入。

#### 允许实现

- 真实模型 Provider 接入
- ToolSandbox 联动

#### 必须通过的测试

- Mock 回归测试不能退化
- Provider 失败必须标准化返回

## 12. 首版不实现内容清单

为避免 AI coder 过度实现，以下能力明确不在 v2 首版范围内：

1. `.team/inbox/` 异步消息收件箱
2. 多进程并发任务认领
3. 强一致消息消费
4. 常驻守护型 Agent
5. 自动记忆压缩与长历史折叠
6. 数据库后端
7. 分布式部署
8. 复杂权限系统
9. 自主规划多个工作流分支并发执行

---

## 13. v3 演进方向

当 v2 首版稳定后，可按需演进：

1. 引入 inbox 机制，支持异步 Agent 协作
2. 支持 daemon lifecycle
3. 支持多任务池
4. 引入可替换存储后端
5. 引入独立 Recorder Agent
6. 支持压缩型对话记忆模块
7. 支持更复杂的 reviewer/critic 多轮博弈

## 14. 最终结论

v2 版本的目标不是构建一个完整的通用智能体平台，而是定义一个**边界清晰、可闭环、可测试、可扩展的多 Agent 协作框架首版**。

本版本通过以下方式收敛复杂度：

- 统一采用半中心化控制
- 用文件系统作为状态真源
- 将 Agent 收敛为单次调用模式
- 将 Recorder 收敛为系统级审计能力
- 将 Memory/Compression 降级为后续能力
- 将 TDD 升级为 AI coder 必须遵守的阶段契约

在此基础上，AI coder 可以按阶段稳定实现，系统也保留了后续从“首版流程框架”向“更通用自治多 Agent 框架”演进的空间。
