# 基于 TypeScript 参考实现的 Agent Python 重写概要设计

## 一、文档目的

本文档用于指导基于 `refs` 目录中的 TypeScript 参考实现，重写一个在行为上严格对齐的 Python 版本 Agent 运行时。这里所说的“严格对齐”，不是要求 Python 版本保留相同的语法结构，而是要求在相同输入、相同工具集合、相同中断条件和相同外部回调约束下，Python 版本能够产出与参考实现一致的消息序列、事件顺序、状态变化、终止条件以及错误表现。

本设计面向工程实现，不追求概念化描述，而是明确哪些语义属于必须保留的核心约束，哪些属于可延后实现的平台增强能力。后续 Python 代码、测试用例和行为比对，都应以本文档为准。

## 二、设计范围

本次概要设计覆盖以下内容：Agent 对象的职责与状态模型、`prompt` 与 `continue` 两类入口语义、主循环的推进逻辑、流式 assistant 消息的构造与收敛方式、工具调用的完整流水线、steering 与 follow-up 队列的处理规则、事件发射顺序、异常与中止语义，以及代理流式协议的兼容要求。

本次设计不要求在第一阶段完整实现所有平台参数，但会明确哪些参数属于严格对齐的一部分，哪些参数可以先保留接口后补充落地。

## 三、设计原则

Python 版本的首要目标是行为一致，而不是接口表面相似。只要某个 TypeScript 细节会影响外部观察到的结果，就应视为必须保留的语义。这里的“外部观察结果”包括消息何时进入上下文、事件何时触发、工具何时被认为正在执行、错误是以抛异常还是以协议消息返回，以及一次运行何时才真正结束。

设计上应遵循三个原则。第一，所有与会话推进相关的逻辑都应收束在 Agent 生命周期与主循环中，避免出现“部分状态在 Agent 内，部分状态散落在工具或调用侧”的分裂实现。第二，所有失败尽量编码为协议内可消费的结果，而不是直接让运行过程崩溃。第三，状态更新必须与事件发射顺序保持一致，因为外部 UI、日志系统和上层调度逻辑都依赖这个顺序。

## 四、参考实现中的核心对象与职责

参考实现中的 `Agent` 是一个有状态包装器。它不直接定义模型推理逻辑，而是持有当前会话状态、工具列表、回调钩子、排队消息和订阅监听器，并通过低层 `runAgentLoop` 或 `runAgentLoopContinue` 驱动一次完整运行。Python 版本也应保持这一职责划分：高层 `Agent` 负责状态管理与生命周期，低层循环负责推进消息流与工具执行。

`AgentState` 在语义上至少包含以下内容：系统提示词、当前模型、思考等级、工具数组、完整消息数组、当前是否处于运行中、当前流式中的 assistant 快照、尚未结束的工具调用标识集合，以及最近一次失败的错误信息。这里面最容易在重写时被弱化的是 `streamingMessage` 与 `pendingToolCalls`，但这两个字段是运行态可观测性的关键组成部分，不能省略。

另外，参考实现显式区分“公开状态”和“内部可变状态”。这意味着 Python 版本也不应把上下文列表直接裸露给外部随意修改，而应通过 Agent 自身的运行入口和状态更新流程维护一致性。

## 五、运行入口设计

Python 版本需要保留两个正式入口，即 `prompt(...)` 与 `continue()`。

`prompt(...)` 的语义是：接收新的用户输入或消息对象，将其规范化为内部消息列表，随后以“新增输入启动运行”的方式开始一次新的 Agent 任务。如果当前已有活动运行，则必须拒绝再次启动，而不是隐式排队或覆盖当前运行。

`continue()` 的语义更严格。它表示“不新增输入，基于当前 transcript 再推进一次”。参考实现要求：若当前没有任何消息，则直接报错；若最后一条消息不是 assistant，则允许继续；若最后一条消息是 assistant，则不能简单继续调用模型，而是要先尝试消费 steering 队列，如果有 steering 消息，则把这些消息当作新的 prompt 重新进入运行；若 steering 为空，再尝试 follow-up 队列；只有两个队列都为空时，才抛出“不能从 assistant 继续”的错误。这个分支是严格对齐的一部分，不能在 Python 版本中省略。

## 六、生命周期设计

一次运行开始时，Agent 必须创建独立的活动运行对象，其中至少包含一个 `AbortController` 等价物、一个代表“任务彻底结束”的 promise 或 future，以及与之关联的状态清理逻辑。运行开始后，状态中的 `isStreaming` 必须设置为真，`streamingMessage` 清空，`errorMessage` 清空。

需要特别注意的是，参考实现中“发出了 `agent_end` 事件”并不代表运行已经彻底结束。真正的结束时刻是：当前运行的所有事件监听器都按注册顺序执行并 `await` 完成，`finishRun()` 完成清理，活动运行的 promise 被 resolve，外部通过 `waitForIdle()` 才能观察到 idle。Python 版本必须保留这个时序，否则会出现监听器尚未处理完但调用方已经认为任务结束的语义偏差。

运行结束后的统一清理内容包括：`isStreaming` 置为假，`streamingMessage` 清空，`pendingToolCalls` 置空集合，并清除活动运行标识。

## 七、消息模型与上下文边界

参考实现使用 `AgentMessage` 作为运行时内部统一消息抽象，其中既包括标准的 user、assistant、toolResult 消息，也允许扩展型自定义消息。Python 版本也应采用“内部消息模型”和“模型接口消息模型”分离的设计。

模型调用前必须经历两个阶段。第一阶段是 `transformContext`，它作用在内部消息层，主要用于裁剪上下文、注入额外消息或做其他会话级变换。第二阶段是 `convertToLlm`，它把内部消息转换成模型真正能够接受的消息结构，并过滤掉不应发送给模型的消息。重要的是，这两个步骤不是“每次运行只做一次”，而是“每轮调用模型前都重新做一次”。这意味着只要上下文在一轮中发生了变化，例如新增了工具结果或插入消息，下一轮模型调用就必须重新变换和转换。

## 八、主循环设计

主循环采用“两层循环”的结构，这不是实现细节，而是语义本身。

内层循环负责推进当前任务。每次迭代时，先在必要时发出 `turn_start`，然后处理 pending 消息，也就是待注入的 steering 或 follow-up 消息。注入时，每条消息都必须发出 `message_start` 和 `message_end` 事件，并同时加入当前上下文和本次运行新增消息列表。之后再发起一次 assistant 流式回复，拿到最终 assistant 消息后判断其停止原因和工具调用情况。

如果本轮 assistant 消息中包含 toolCall，则执行工具，并将所有 toolResult 追加回上下文和新增消息列表，然后结束本 turn，并继续下一轮。每个 turn 结束时都要重新读取 steering 队列，因为运行过程中外部可能又插入了新的 steering 消息。

外层循环只处理一种情况：当内层循环看起来已经没有工具调用且没有 pending 消息时，Agent 理论上可以结束，但在真正结束前还必须读取 follow-up 队列。如果 follow-up 队列非空，则这些消息会被转成新的 pending 消息，重新进入内层循环继续执行。只有在“没有工具调用、没有 steering、没有 follow-up”三个条件同时满足时，整次 Agent 运行才可以发出 `agent_end` 并结束。

这部分逻辑的含义可以概括为：steering 是“中途插队，下一轮立刻生效”，follow-up 是“本来要停下时再接着干”，两者不能混用，也不能由一个统一队列代替。

## 九、流式 assistant 消息设计

assistant 回复必须按“流式快照逐步成形，最终结果整体收敛”的方式实现。

在收到流的 `start` 事件时，应创建 partial assistant message，并立即将其加入上下文，同时发出 `message_start`。从这一刻开始，这条 partial message 就是当前上下文中的最后一条 assistant 消息。

在收到文本增量、thinking 增量或 toolCall 增量事件时，不应新增消息，而应更新这条 partial message 的内容，并用更新后的快照覆盖上下文中的最后一条 assistant 消息，同时发出 `message_update`。因此，partial message 本质上是“当前已生成内容的完整快照”，而不是单独的增量片段。

在收到 `done` 或 `error` 事件时，应通过底层流对象拿到 final assistant message，并使用 final 覆盖上下文中的 partial message。如果此前从未收到 `start` 事件，则需要直接把 final 作为新消息插入上下文并补发 `message_start`。无论是哪种情况，都必须在 final 落地后发出 `message_end`。

另外还要保留一个兜底分支：即使事件迭代自然结束，没有显式进入 `done/error` 分支，仍然要再次调用底层 `result()` 方法取得最终消息，并用同样规则落地。这是参考实现的一部分，用于保证流协议在边界情况下也能收敛出最终消息。

## 十、工具调用设计

工具调用处理必须保留完整流水线，且每个阶段的错误语义要与参考实现一致。

当 assistant 消息中包含一个或多个 `toolCall` 内容块时，系统首先提取这些工具调用，并根据配置选择顺序执行或并行执行。无论采用哪种模式，每个工具调用在正式执行前都必须先发出 `tool_execution_start` 事件。

工具调用的准备阶段包含四步。第一步，根据工具名在当前工具集合中查找目标工具；找不到工具时，不抛异常，而是立即构造一个错误的工具结果。第二步，若工具提供 `prepareArguments`，则先用它对原始参数进行兼容性预处理。第三步，对预处理后的参数执行 schema 校验。第四步，若存在 `beforeToolCall`，则调用该钩子；若钩子返回阻断标志，则本次工具调用不执行，直接构造错误工具结果。

只有通过上述准备阶段的工具调用，才进入真正的 execute 阶段。execute 过程中工具可以持续通过回调上报中间结果，运行时需要把这些中间结果转换为 `tool_execution_update` 事件发出。需要注意，参考实现会等待这些 update 事件全部完成后，再返回最终工具结果或错误结果，因此 Python 版本也应保证 update 事件不会在最终结果落地之后“拖尾”出现。

工具执行结束后，若实现了 `afterToolCall`，则必须调用该钩子。该钩子不是对结果做深度 merge，而是字段级覆盖：若返回了新的 `content`，则整体替换原 content；若返回了新的 `details`，则整体替换原 details；若返回了新的 `isError`，则覆盖原有错误标记。Python 版本必须按这个规则实现，不能做字典递归合并。

最终，系统先发出 `tool_execution_end`，再构造 `toolResult` 消息，并对该消息发出 `message_start` 与 `message_end`，随后将其加入上下文与新增消息列表。这一顺序也属于严格对齐内容。

## 十一、顺序执行与并行执行语义

顺序执行模式较直接，即按 assistant 消息中的工具调用顺序，一个个完成准备、执行、后处理和结果落地，前一个完成后才处理下一个。

并行执行模式有一个容易实现错的关键点：它不是从一开始就“所有工具完全并发”。参考实现的语义是，先按顺序做每个工具调用的准备阶段，也就是工具查找、参数预处理、参数校验和 before hook；在这个阶段就已经能确定哪些工具立即失败、哪些工具允许执行。只有那些允许执行的工具，才会被并发提交到 execute 阶段。随后在结果收集和后处理时，仍然按 assistant 原始工具顺序逐个 finalize。换句话说，并行的只有 execute，本次 assistant turn 的可观察结果顺序仍然稳定。

因此，Python 版本不能简单采用 `asyncio.as_completed()` 后按完成顺序落地结果，否则会破坏 toolResult 顺序与事件顺序。

## 十二、队列设计

steering 队列和 follow-up 队列都需要实现，并且都应支持两种 drain 模式：`all` 与 `one-at-a-time`。在 `all` 模式下，每次 drain 返回当前队列中的全部消息；在 `one-at-a-time` 模式下，每次只返回第一条消息。这个细节会影响运行过程中的上下文注入节奏，属于参考实现显式提供的行为，不应忽略。

steering 队列用于运行中插入下一轮立即生效的消息。系统在运行开始前会先 poll 一次 steering 队列，用来接收“用户在等待期间已经输入的新消息”；而在每个 turn 结束后也会再次 poll steering 队列，用于处理中途新插入的指令。

follow-up 队列只在 Agent 原本准备结束时读取。如果存在 follow-up，则并不立刻额外开一个独立运行，而是把 follow-up 变成 pending 消息，继续当前这次 Agent 运行。

## 十三、事件与内部状态同步

Python 版本应保持“先归约内部状态，再通知监听器”的处理顺序。也就是说，收到事件后，Agent 先根据事件更新内部状态，再依次调用所有订阅监听器，并等待监听器完成。

其中，`message_start` 和 `message_update` 事件会更新 `streamingMessage`，`message_end` 会清空 `streamingMessage` 并把消息加入状态中的完整消息列表。`tool_execution_start` 会把对应 `toolCallId` 放入 `pendingToolCalls`，`tool_execution_end` 会将其移除。`turn_end` 如果看到 assistant 消息带有 `errorMessage`，则更新状态中的最近错误。`agent_end` 到来时则清空 `streamingMessage`。

如果监听器是在没有活动运行的情况下被调用，应视为运行时错误，因为参考实现假定所有事件都发生在活跃运行上下文中。

## 十四、异常与中止设计

参考实现倾向于将错误编码为消息或结果，而不是把错误直接抛出到循环外层。工具未找到、参数非法、执行失败，都会被转换成 `isError=true` 的 `toolResult`。模型调用失败或流被中止，则体现在 assistant final message 的 `stopReason=error` 或 `stopReason=aborted`，并带上 `errorMessage`。

如果出现真正绕过上述协议的运行时异常，例如某个内部逻辑抛出未处理错误，则外层生命周期包装器需要构造一条失败 assistant 消息，写入 transcript，更新 `errorMessage`，并发出 `agent_end`，从而让系统以可恢复的方式收尾。这样设计的目的，是确保即使底层实现异常，外层也仍然能观察到“本次运行失败且已结束”的明确状态，而不是永远挂在运行中。

中止语义也要严格保留。Agent 的 `abort()` 应只针对当前活动运行生效。中止信号需要一路传递到模型流和工具执行。对于代理流式实现，还需要在中止时取消底层 reader，避免连接悬挂。

## 十五、代理流式协议兼容要求

如果 Python 第一阶段走的是本地模型 SDK，可以暂不实现代理流式协议；但只要需要兼容 `proxy.ts` 的语义，就必须完整支持服务端发来的事件重建逻辑。

该协议的关键点在于：服务端只发事件增量，客户端负责维护一条 partial assistant message；文本与 thinking 通过字符串累积完成；toolCall 的参数通过增量 JSON 文本拼接后渐进解析；当流结束时，客户端根据 `done` 或 `error` 事件补齐 `stopReason`、`usage` 和 `errorMessage`，并把当前 partial 作为最终消息输出。

同时，读取流时需要处理分块传输与缓冲区拼接，只消费以 `data:` 开头的有效事件行；网络错误、代理错误和中止都要转换成 assistant 错误消息，而不是直接把异常抛给上层。

## 十六、Python 版本的建议模块划分

为了保持语义清晰，建议 Python 版本至少划分为四层。第一层是类型与消息模型层，定义 `AgentMessage`、`AssistantMessage`、`ToolResultMessage`、`AgentEvent` 等结构。第二层是 Agent 生命周期层，负责公开 API、状态管理、事件监听和运行包装。第三层是循环推进层，负责 `run_agent_loop`、`run_agent_loop_continue` 和双层循环主体。第四层是流式与工具适配层，分别处理模型流协议和工具执行流水线。

若后续还需要兼容代理协议，可单独增加 proxy stream 模块，避免把代理解析逻辑混入核心循环。

## 十七、最小可落地实现建议

第一阶段建议优先完成以下能力：`prompt` 与 `continue` 两类入口、单活动运行保护、双层主循环、partial/final assistant 收敛、顺序与并行工具执行、双队列机制、事件与状态同步、协议化错误收尾。只要这些能力正确，Python 版本就已经具备与 TypeScript 版本做行为比对的基础。

第二阶段再补充：动态 API Key、平台透传参数、代理流式协议、扩展消息类型以及更细粒度的性能优化。

## 十八、验收标准

Python 重写完成后，验收不应只看“能跑”，而应按行为比对。至少需要验证以下方面：相同输入下的消息追加顺序是否一致，`turn_start/turn_end` 与 `message_start/update/end` 顺序是否一致，工具并行模式下结果顺序是否稳定，steering 与 follow-up 的触发时机是否一致，错误与中止是否被编码成协议消息，以及 `wait_for_idle()` 是否在所有监听器结束后才返回。

只有当这些行为都能对齐时，才可以认为 Python 版本在语义上真正继承了 TypeScript 参考实现。
