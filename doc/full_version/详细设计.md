## 一、文档目标

本文档是 [概要设计.md](/Users/biguncle/project/uu-work/doc/概要设计.md) 的细化版本，面向后续 Python 工程实现。本文档的目标不是重复概要设计中已经明确的约束，而是在其基础上补齐可直接指导编码落地的内容，包括代码目录规划、模块边界、核心类与函数的职责、关键状态流转、主要接口契约，以及覆盖关键语义的功能逻辑伪代码。

本次详细设计的代码根目录明确为 `agent_core`。因此，下文中涉及的目录规划、文件划分和模块引用关系，均以 `agent_core` 为根目录展开。设计目标仍然是“严格对齐 TypeScript 参考实现的运行语义”，而不是仅实现一个功能上大致相似的 Agent。

## 二、实施范围与交付边界

本次详细设计覆盖一个完整的 Python Agent Runtime 实现蓝图。它包括：基础消息模型、Agent 状态与事件系统、生命周期与运行句柄、主循环、流式 assistant 响应处理、工具执行流水线、双队列机制、模型适配器接口、代理流式协议兼容层、以及最小化的测试目录建议。

本次详细设计不要求第一阶段必须实现所有外部 provider 适配，也不要求在第一阶段实现复杂的上下文压缩策略。但这些能力会在接口层预留明确扩展点，保证后续实现时无需返工核心运行时。

## 三、总体实现策略

从工程组织上看，建议在 `agent_core` 下建立一套独立、语义完整的 runtime 包，而不是把所有职责堆在单一 `agent.py` 文件中。原因有两个。第一，参考实现中的语义层次本身就比较清晰，生命周期、流式处理、工具执行和状态同步分别承担不同责任，Python 版本如果混写在一起，会让后续对齐和测试变得困难。第二，运行时的严格语义很依赖可单测的模块边界，例如“只测工具并行顺序”“只测 partial/final 收敛”“只测 continue 的边界条件”，这些都要求模块划分足够明确。

因此，建议采用“运行时核心 + 适配层 + 测试层”三段式结构。运行时核心实现完全不依赖具体 provider 的协议细节，只依赖统一的流式接口与工具接口。适配层用于把 OpenAI、本地包装器或代理 SSE 对接到运行时。测试层则围绕行为对齐做用例设计。

## 五、模块职责设计

### 5.1 `agent_core/runtime/models.py`

该模块负责定义运行时内部的数据结构。这里不建议用松散字典到处传递，而应采用 `dataclass`、`TypedDict` 或 `Protocol` 的组合，至少把“内部消息”和“模型输入消息”区分开来。

该模块应定义以下核心结构：文本内容块、图片内容块、thinking 内容块、toolCall 内容块、user 消息、assistant 消息、toolResult 消息、AgentToolResult、AgentContext、ToolExecutionMode、ThinkingLevel，以及运行时通用的 `AgentMessage` 联合类型。对于 `AssistantMessage`，需要保留 `stop_reason`、`error_message`、`usage`、`provider`、`model`、`api`、`timestamp` 等字段，因为它们在 final 消息收敛时会参与状态同步与错误判断。

该模块还应定义 `AgentTool` 的 Python 侧协议。建议用 `Protocol` 表达，要求工具对象至少包含 `name`、`label`、可选 `prepare_arguments`、以及 `async execute(...)` 方法。工具执行返回值应统一包装为 `AgentToolResult`，而不是允许工具直接返回任意结构。

### 5.2 `agent_core/runtime/events.py`

该模块负责定义运行时事件模型和监听器签名。事件类型应完整覆盖 `agent_start`、`agent_end`、`turn_start`、`turn_end`、`message_start`、`message_update`、`message_end`、`tool_execution_start`、`tool_execution_update`、`tool_execution_end`。不建议将这些事件仅用字符串和任意字典拼装，而应为每种事件定义清晰的数据结构，以便状态归约和测试断言。

监听器签名建议统一为异步函数，形式类似 `async def listener(event, cancel_token) -> None`。即使允许同步监听器，也应在 Agent 内部统一包装为 awaitable，保持事件处理语义一致。

### 5.3 `agent_core/runtime/state.py`

该模块负责定义 Agent 的公开状态对象和内部可变状态对象。建议拆成 `AgentStateSnapshot` 与 `MutableAgentState` 两类。前者用于对外只读暴露，后者用于内部更新。需要重点承载的状态包括：系统提示词、当前模型配置、thinking level、tools、messages、is_streaming、streaming_message、pending_tool_calls、error_message。

由于参考实现要求“对外暴露的工具列表和消息列表不应直接引用原始可变数组”，因此 Python 版本也应通过复制或只读包装暴露这些集合，避免外部绕过 Agent 主流程修改状态。

### 5.4 `agent_core/runtime/queues.py`

该模块负责封装 steering 队列与 follow-up 队列的共性逻辑。建议提供 `PendingMessageQueue` 类，内部维护消息列表与模式字段，支持 `enqueue()`、`drain()`、`clear()`、`has_items()`。`drain()` 必须实现 `all` 和 `one-at-a-time` 两种策略，因为这不是性能细节，而是外部可观察行为。

### 5.5 `agent_core/runtime/run_context.py`

该模块负责定义运行期句柄，例如 `RunHandle`、`CancelToken`、`RunFuture`。这里建议不要直接把 `asyncio.Task.cancel()` 作为唯一中止手段，而是引入一个显式取消信号对象，例如基于 `asyncio.Event` 的 `CancelToken`。模型流、工具执行、监听器调度和代理 reader 都应能观察这个取消信号。

`RunHandle` 应至少包含：取消信号、idle future、完成回调、运行标识。这样实现 `abort()`、`wait_for_idle()` 和运行结束清理时会更稳定。

### 5.6 `agent_core/runtime/config.py`

该模块定义 `AgentOptions`、`AgentLoopConfig` 以及相关 hook 类型。它主要承载运行期配置而不是业务状态。建议把下列接口集中定义在这里：`convert_to_llm`、`transform_context`、`get_api_key`、`before_tool_call`、`after_tool_call`、`get_steering_messages`、`get_followup_messages`、`stream_fn`、`tool_execution`、`thinking_budgets`、`transport`、`session_id`。

设计上应明确区分“构造 Agent 时长期存在的配置”和“每轮运行时从 Agent 状态快照中组装出的 loop config”。这点直接对应 TypeScript 参考实现中的 `createLoopConfig()`。

### 5.7 `agent_core/runtime/errors.py`

该模块集中定义运行时异常。建议至少定义：`AgentRuntimeError`、`AgentAlreadyRunningError`、`InvalidContinuationError`、`ListenerOutsideRunError`、`ToolValidationError`、`ProxyProtocolError`。虽然大多数失败最终会被包装成协议消息，但内部仍需要明确异常类型，方便日志与单测区分。

### 5.8 `agent_core/runtime/stream_handler.py`

该模块负责处理一次 assistant 流式响应，将模型流协议转换为运行时的消息与事件。它的职责是：在收到流 `start` 时创建 partial assistant message；在增量事件期间持续更新该 partial；在 `done` 或 `error` 时拿到 final message 并完成收敛；在任何合法分支结束后返回 final assistant message。

这里不建议把消息更新逻辑直接塞到主循环里，否则流式处理与 turn 推进会耦合过深，不利于测试。`stream_handler.py` 应暴露一个单独函数，例如 `async stream_assistant_response(...) -> AssistantMessage`。

### 5.9 `agent_core/runtime/tool_executor.py`

该模块负责 assistant 消息中工具调用的提取、准备、执行、后处理和结果落地。它需要暴露 `execute_tool_calls()` 总入口，并将顺序执行与并行执行拆为两个实现函数。工具准备、工具执行、结果 finalize 也建议进一步拆为独立函数，因为这些是最容易出语义偏差的部分。

其中，工具执行并行模式必须保留“准备阶段串行、执行阶段并发、结果落地按原顺序”的精确语义。该模块是严格对齐的关键模块之一，建议单独编写较完整的单元测试。

### 5.10 `agent_core/runtime/loop.py`

该模块负责实现 `run_agent_loop()`、`run_agent_loop_continue()` 和共享的 `run_loop()`。它承接 `agent.py` 传入的上下文快照、配置与事件发射器，并驱动完整的双层循环。它本身不维护 Agent 状态，只操作运行期上下文和当前新增消息列表。

设计上，`loop.py` 不应感知外部 UI，不应直接改动 Agent 对象，只应通过 emit 回调对外发出事件。这与 TypeScript 参考实现的职责划分保持一致。

### 5.11 `agent_core/runtime/proxy_stream.py`

该模块负责在需要兼容代理协议时，把代理 SSE 增量事件重建为统一的 assistant 流式事件流。其内部需要处理字节流读取、行缓冲区拼接、`data:` 前缀过滤、增量 JSON 累积解析、error 与 aborted 的收敛逻辑。

如果第一阶段尚不启用代理协议，该模块可以先实现接口与测试桩，但目录上仍建议保留，以便后续功能无缝补齐。

### 5.12 `agent_core/runtime/agent.py`

该模块提供对外 Agent 类，是整个运行时的唯一正式入口。其主要职责包括：保存长期状态、注册监听器、创建运行句柄、暴露 `prompt()`、`continue_()`、`abort()`、`wait_for_idle()`、`steer()`、`follow_up()`、`clear_*_queue()` 等方法，在运行开始前组装上下文快照与 loop config，在运行过程中归约状态并分发事件，在运行失败时构造兜底 assistant 错误消息，并在结束时完成状态清理。

该模块不负责模型流协议解析，也不负责工具执行细节，它只负责 orchestration。

### 5.13 `agent_core/adapters/llm_protocol.py`

该模块定义对模型层的最小协议抽象。建议定义一个 `StreamFnProtocol` 或 `LLMStreamClientProtocol`，要求其提供 `async stream(model, context, options) -> AsyncMessageEventStream` 之类的统一入口。这样运行时可以不关心底层到底是 OpenAI SDK、内部包装器，还是走代理 HTTP。

### 5.14 `agent_core/adapters/openai_adapter.py`

该模块用于将具体模型 SDK 封装为运行时可消费的流式接口。由于当前工程里未有成熟的 strict-runtime 实现，因此建议该模块只承担 provider 交互和协议适配，不掺杂主循环语义。

### 5.15 `agent_core/adapters/proxy_adapter.py`

该模块负责发起代理 HTTP 请求，并将字节流交给 `proxy_stream.py` 处理。它的职责重点是认证、请求参数组装和 reader 生命周期控制。

### 5.16 `agent_core/utils/streaming_json.py`

该模块提供增量 JSON 拼接与渐进解析工具，主要用于代理协议中的 toolCall 参数构造。这里应保证“JSON 尚未完整时返回部分可解析结果或空对象，而不是抛致命错误”，以对齐参考实现的 streaming JSON 行为。

## 六、核心类与接口设计

### 6.1 `Agent` 类

`Agent` 类建议提供以下公开方法：`prompt()`、`continue_()`、`subscribe()`、`abort()`、`wait_for_idle()`、`steer()`、`follow_up()`、`clear_steering_queue()`、`clear_followup_queue()`、`clear_all_queues()`、`reset()`。其中 `continue` 是 Python 关键字冲突，建议对外命名为 `continue_()`，文档层和调用习惯中仍可视为“continue 语义”。

`Agent` 内部应持有：状态对象、监听器列表、steering 队列、follow-up 队列、活动运行句柄、长期配置项。与概要设计一致，活动运行期间禁止再次启动 prompt 或 continue。

### 6.2 `PendingMessageQueue` 类

该类的职责很纯粹，只做消息入队与按模式出队，不应混入业务判断。它的好处是 steering 和 follow-up 可以共享相同实现，避免在 Agent 内部写两套 drain 逻辑。

### 6.3 `CancelToken` 类

该类建议基于 `asyncio.Event` 实现，并提供 `cancel()`、`is_cancelled()`、`raise_if_cancelled()`、`wait_cancelled()` 等方法。原因是 Python 环境里不能简单照搬 JS 的 `AbortSignal`，但运行时语义要求“流处理、工具执行、事件监听”都能统一观察取消状态。

### 6.4 `AsyncMessageEventStream` 接口

该接口应抽象出模型流对象的最小能力，包括异步迭代事件和读取最终结果。它至少需要支持两种操作：`async for event in stream` 和 `final = await stream.result()`。这是 strict alignment 的关键，因为 `stream_handler` 在某些分支中需要提前拿 final，也需要在流自然结束后再调用一次 `result()` 做兜底。

## 七、关键功能设计与伪代码

### 7.1 Agent 生命周期与公开入口

这部分的实现目标，是在不破坏外部可观察行为的前提下，将 prompt、continue 和运行句柄组织起来。

伪代码如下：

```python
class Agent:
    async def prompt(self, input_value, images=None):
        if self._active_run is not None:
            raise AgentAlreadyRunningError()
        prompt_messages = self._normalize_prompt_input(input_value, images)
        await self._run_prompt_messages(prompt_messages)

    async def continue_(self):
        if self._active_run is not None:
            raise AgentAlreadyRunningError()

        last_message = self._state.messages[-1] if self._state.messages else None
        if last_message is None:
            raise InvalidContinuationError("no messages to continue from")

        if last_message.role == "assistant":
            steering = self._steering_queue.drain()
            if steering:
                await self._run_prompt_messages(steering, skip_initial_steering_poll=True)
                return

            followups = self._followup_queue.drain()
            if followups:
                await self._run_prompt_messages(followups, skip_initial_steering_poll=False)
                return

            raise InvalidContinuationError("cannot continue from assistant")

        await self._run_continuation()
```

### 7.2 运行包装与 idle 语义

这一部分的重点不是启动任务，而是保证“运行完成”和“监听器处理完成”是同一个完成点。

伪代码如下：

```python
async def _run_with_lifecycle(self, executor):
    if self._active_run is not None:
        raise AgentAlreadyRunningError()

    cancel_token = CancelToken()
    idle_future = asyncio.get_running_loop().create_future()
    self._active_run = RunHandle(cancel_token=cancel_token, idle_future=idle_future)

    self._state.is_streaming = True
    self._state.streaming_message = None
    self._state.error_message = None

    try:
        await executor(cancel_token)
    except Exception as exc:
        await self._handle_run_failure(exc, cancelled=cancel_token.is_cancelled())
    finally:
        self._finish_run()

def _finish_run(self):
    self._state.is_streaming = False
    self._state.streaming_message = None
    self._state.pending_tool_calls = set()
    self._active_run.idle_future.set_result(None)
    self._active_run = None
```

### 7.3 主循环

主循环必须保留双层循环结构，因为这是 strict alignment 的核心。

伪代码如下：

```python
async def run_loop(current_context, new_messages, config, emit, cancel_token):
    first_turn = True
    pending_messages = await config.get_steering_messages() or []

    while True:
        has_more_tool_calls = True

        while has_more_tool_calls or pending_messages:
            if not first_turn:
                await emit(TurnStartEvent())
            else:
                first_turn = False

            for message in pending_messages:
                await emit(MessageStartEvent(message=message))
                await emit(MessageEndEvent(message=message))
                current_context.messages.append(message)
                new_messages.append(message)
            pending_messages = []

            assistant_message = await stream_assistant_response(
                context=current_context,
                config=config,
                emit=emit,
                cancel_token=cancel_token,
            )
            new_messages.append(assistant_message)

            if assistant_message.stop_reason in ("error", "aborted"):
                await emit(TurnEndEvent(message=assistant_message, tool_results=[]))
                await emit(AgentEndEvent(messages=new_messages))
                return

            tool_calls = extract_tool_calls(assistant_message)
            has_more_tool_calls = len(tool_calls) > 0

            tool_results = []
            if has_more_tool_calls:
                tool_results = await execute_tool_calls(
                    current_context=current_context,
                    assistant_message=assistant_message,
                    config=config,
                    emit=emit,
                    cancel_token=cancel_token,
                )
                for result in tool_results:
                    current_context.messages.append(result)
                    new_messages.append(result)

            await emit(TurnEndEvent(message=assistant_message, tool_results=tool_results))
            pending_messages = await config.get_steering_messages() or []

        followups = await config.get_followup_messages() or []
        if followups:
            pending_messages = followups
            continue

        break

    await emit(AgentEndEvent(messages=new_messages))
```

### 7.4 assistant 流式响应处理

这一部分的目标是确保 partial 与 final 的收敛规则精确一致。

伪代码如下：

```python
async def stream_assistant_response(context, config, emit, cancel_token):
    messages = context.messages
    if config.transform_context is not None:
        messages = await config.transform_context(messages, cancel_token)

    llm_messages = await config.convert_to_llm(messages)
    llm_context = build_llm_context(system_prompt=context.system_prompt, messages=llm_messages, tools=context.tools)

    api_key = await resolve_api_key(config)
    stream = await config.stream_fn(config.model, llm_context, config.with_api_key(api_key), cancel_token)

    partial_message = None
    added_partial = False

    async for event in stream:
        if event.type == "start":
            partial_message = event.partial
            context.messages.append(partial_message)
            added_partial = True
            await emit(MessageStartEvent(message=copy_message(partial_message)))
            continue

        if event.type in STREAM_UPDATE_EVENTS:
            partial_message = event.partial
            context.messages[-1] = partial_message
            await emit(MessageUpdateEvent(message=copy_message(partial_message), assistant_event=event))
            continue

        if event.type in ("done", "error"):
            final_message = await stream.result()
            if added_partial:
                context.messages[-1] = final_message
            else:
                context.messages.append(final_message)
                await emit(MessageStartEvent(message=copy_message(final_message)))
            await emit(MessageEndEvent(message=final_message))
            return final_message

    final_message = await stream.result()
    if added_partial:
        context.messages[-1] = final_message
    else:
        context.messages.append(final_message)
        await emit(MessageStartEvent(message=copy_message(final_message)))
    await emit(MessageEndEvent(message=final_message))
    return final_message
```

### 7.5 工具执行总入口

该部分的职责是根据模式分派到顺序或并行执行，并保证最终返回值都是 `toolResult` 消息列表。

```python
async def execute_tool_calls(current_context, assistant_message, config, emit, cancel_token):
    tool_calls = [item for item in assistant_message.content if item.type == "toolCall"]
    if config.tool_execution == "sequential":
        return await execute_tool_calls_sequential(

    return await execute_tool_calls_parallel(
        current_context, assistant_message, tool_calls, config, emit, cancel_token

```

### 7.6 工具准备阶段

工具准备阶段是 strict alignment 里最关键的边界之一，因为找不到工具、参数校验失败和 before hook 阻断，全部都必须转化为“立即失败的工具结果”，而不是终止整个循环。

```python
async def prepare_tool_call(current_context, assistant_message, tool_call, config, cancel_token):
    tool = find_tool_by_name(current_context.tools, tool_call.name)
    if tool is None:
        return ImmediateToolOutcome(result=create_error_tool_result(f"Tool {tool_call.name} not found"), is_error=True)

    try:
        prepared_tool_call = maybe_prepare_arguments(tool, tool_call)
        validated_args = validate_tool_arguments(tool, prepared_tool_call)

        if config.before_tool_call is not None:
            before_result = await config.before_tool_call(
                assistant_message=assistant_message,
                tool_call=tool_call,
                args=validated_args,
                context=current_context,
                cancel_token=cancel_token,
            )
            if before_result and before_result.block:
                return ImmediateToolOutcome(
                    result=create_error_tool_result(before_result.reason or "Tool execution was blocked"),
                    is_error=True,
                )

        return PreparedToolCall(tool_call=tool_call, tool=tool, args=validated_args)
    except Exception as exc:
        return ImmediateToolOutcome(result=create_error_tool_result(str(exc)), is_error=True)
```

### 7.7 工具顺序执行

```python
async def execute_tool_calls_sequential(current_context, assistant_message, tool_calls, config, emit, cancel_token):
    results = []
    for tool_call in tool_calls:
        await emit(ToolExecutionStartEvent(tool_call_id=tool_call.id, tool_name=tool_call.name, args=tool_call.arguments))
        preparation = await prepare_tool_call(current_context, assistant_message, tool_call, config, cancel_token)

        if preparation.kind == "immediate":
            tool_result = await emit_tool_call_outcome(tool_call, preparation.result, preparation.is_error, emit)
            results.append(tool_result)
            continue

        executed = await execute_prepared_tool_call(preparation, emit, cancel_token)
        tool_result = await finalize_executed_tool_call(
            current_context, assistant_message, preparation, executed, config, emit, cancel_token
        )
        results.append(tool_result)

    return results
```

### 7.8 工具并行执行

这里必须强调，不能按完成顺序收集结果，而要按 assistant 原调用顺序 finalize。

```python
async def execute_tool_calls_parallel(current_context, assistant_message, tool_calls, config, emit, cancel_token):
    results = []
    runnable_calls = []

    for tool_call in tool_calls:
        await emit(ToolExecutionStartEvent(tool_call_id=tool_call.id, tool_name=tool_call.name, args=tool_call.arguments))
        preparation = await prepare_tool_call(current_context, assistant_message, tool_call, config, cancel_token)
        if preparation.kind == "immediate":
            tool_result = await emit_tool_call_outcome(tool_call, preparation.result, preparation.is_error, emit)
            results.append(tool_result)
        else:
            runnable_calls.append(preparation)

    running_tasks = [
        asyncio.create_task(execute_prepared_tool_call(prepared, emit, cancel_token))
        for prepared in runnable_calls
    ]

    for prepared, task in zip(runnable_calls, running_tasks):
        executed = await task
        tool_result = await finalize_executed_tool_call(
            current_context, assistant_message, prepared, executed, config, emit, cancel_token
        )
        results.append(tool_result)

    return results
```

### 7.9 工具执行与 update 事件

```python
async def execute_prepared_tool_call(prepared, emit, cancel_token):
    update_tasks = []

    def on_update(partial_result):
        update_tasks.append(asyncio.create_task(
            emit(ToolExecutionUpdateEvent(
                tool_call_id=prepared.tool_call.id,
                tool_name=prepared.tool_call.name,
                args=prepared.tool_call.arguments,
                partial_result=partial_result,
            ))
        ))

    try:
        result = await prepared.tool.execute(
            tool_call_id=prepared.tool_call.id,
            params=prepared.args,
            cancel_token=cancel_token,
            on_update=on_update,
        )
        if update_tasks:
            await asyncio.gather(*update_tasks)
        return ExecutedToolCallOutcome(result=result, is_error=False)
    except Exception as exc:
        if update_tasks:
            await asyncio.gather(*update_tasks)
        return ExecutedToolCallOutcome(result=create_error_tool_result(str(exc)), is_error=True)
```

### 7.10 工具结果后处理

```python
async def finalize_executed_tool_call(current_context, assistant_message, prepared, executed, config, emit, cancel_token):
    result = executed.result
    is_error = executed.is_error

    if config.after_tool_call is not None:
        override = await config.after_tool_call(
            assistant_message=assistant_message,
            tool_call=prepared.tool_call,
            args=prepared.args,
            result=result,
            is_error=is_error,
            context=current_context,
            cancel_token=cancel_token,
        )
        if override is not None:
            result = AgentToolResult(
                content=override.content if override.content is not None else result.content,
                details=override.details if override.details is not None else result.details,
            )
            is_error = override.is_error if override.is_error is not None else is_error

    return await emit_tool_call_outcome(prepared.tool_call, result, is_error, emit)
```

### 7.11 事件归约与监听器分发

Python 版本必须保留“先更新状态，再调用监听器”的顺序。

```python
async def process_event(self, event):
    if event.type == "message_start":
        self._state.streaming_message = event.message
    elif event.type == "message_update":
        self._state.streaming_message = event.message
    elif event.type == "message_end":
        self._state.streaming_message = None
        self._state.messages.append(event.message)
    elif event.type == "tool_execution_start":
        self._state.pending_tool_calls = set(self._state.pending_tool_calls) | {event.tool_call_id}
    elif event.type == "tool_execution_end":
        pending = set(self._state.pending_tool_calls)
        pending.discard(event.tool_call_id)
        self._state.pending_tool_calls = pending
    elif event.type == "turn_end":
        if getattr(event.message, "error_message", None):
            self._state.error_message = event.message.error_message
    elif event.type == "agent_end":
        self._state.streaming_message = None

    if self._active_run is None:
        raise ListenerOutsideRunError()

    for listener in self._listeners:
        await ensure_awaitable(listener(event, self._active_run.cancel_token))
```

## 八、模型适配设计

运行时不应直接依赖某一个 SDK 的消息结构。建议在适配层完成两件事。第一，把运行时传入的标准上下文组装成 provider 所需请求参数。第二，把 provider 的流式结果转换成运行时统一事件流。这样一来，即使后续从 OpenAI 换成内部网关或自建代理，核心运行时不需要改动。

动态 API Key 获取逻辑建议也放在 loop config 中，由 `stream_assistant_response()` 每轮调用前解析一次，而不是在 Agent 初始化时绑定死。这一点在长运行、多轮 tool 调用场景下尤其重要。

## 九、测试设计建议

测试目录虽然不是本次编码主体，但详细设计中必须给出建议，因为严格对齐的实现无法只靠人工验收。建议至少覆盖以下测试。

`test_agent_lifecycle.py` 负责验证：重复 prompt 报错、abort 行为、wait_for_idle 时机、listener 顺序执行。

`test_tool_executor_sequential.py` 负责验证：顺序执行结果顺序、工具找不到、参数校验失败、before block、after override。

`test_tool_executor_parallel.py` 负责验证：准备阶段串行、执行阶段并发、落地顺序稳定、update 事件先于最终结果收敛。

## 十、分阶段实施建议

第一阶段建议先落地 `models.py`、`events.py`、`state.py`、`queues.py`、`run_context.py`、`config.py`、`stream_handler.py`、`tool_executor.py`、`loop.py` 和 `agent.py`。完成后即可先接一个最简单的本地模型流适配器，用于跑通主循环。

第二阶段再补 `proxy_stream.py`、`proxy_adapter.py` 和更完整的 provider 适配。这样实施的好处是：先验证 strict runtime 语义，再接入复杂的网络协议，问题定位会更清晰。

## 十一、详细设计结论

本详细设计的核心思想是，在 `agent_core` 目录下建立一套职责清晰、边界明确、可单测的 Agent Runtime 结构，让 Python 版本在行为上严格继承 TypeScript 参考实现。后续编码过程中，凡是会影响消息顺序、事件时序、状态收敛、中止表现、工具结果顺序或 continue 边界条件的实现选择，都必须优先服从本设计，而不是为了简化代码牺牲语义一致性。

只有按这种方式实现，Python 版本才不仅是“能工作”，而是“可验证地与参考实现对齐”。
