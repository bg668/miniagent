# 多 Agent 轻量协同框架设计审核稿

## 一、总体结论

该方案具备通用多 Agent 框架的基本雏形，核心方向可行，包括任务拆解、角色协作、状态落盘、产物留痕、工具调用、可测试开发等要素，整体上不是错误方案。参考设计文档中的工作空间拓扑、Task/Message Schema、模块划分与测试章节。

但当前版本仍存在三个明显问题：

1. 抽象层次基本正确，但控制范式不够统一
2. 可扩展框架的关键数据结构偏薄
3. 首版实现范围没有收紧，存在过度实现风险

因此，本方案适合作为“通用多 Agent 框架初稿”继续修订，不适合作为“无需补充即可直接交付 AI coder 编码”的最终设计稿。

---

## 二、问题清单

### 问题 1：控制范式不清，中心调度与 Agent 自治混杂

当前文档一方面将 `MessageBus & TaskManager` 定义为纯 I/O 模块，不包含 AI 逻辑；另一方面又让 Agent 在 idle loop 中自行扫描 inbox、claim task、决定何时恢复工作。这样会导致系统到底由谁推进流程并不明确。

这会带来两个后果：

* 实现者无法判断主控制权在 Orchestrator 还是在 Agent
* 后续调试时，任务推进责任不清，容易出现重复领取、空轮询、状态漂移

**判断：这是当前设计的核心结构性问题。**

---

### 问题 2：Task Schema 过薄，不足以支撑通用协作框架

当前 `Task Node` 仅包含：

* `id`
* `subject`
* `description`
* `status`
* `owner`
* `blockedBy` 

这只够表达“待办任务”，不够表达“协作任务单”。缺少以下关键字段：

* 预期执行角色
* 输入产物引用
* 输出产物引用
* 失败原因
* 重试次数
* 父子任务关系
* 优先级
* 创建/更新时间

如果不补这些字段，后续 Reviewer、Recorder、Dispatcher 都要依赖额外隐含约定，框架会迅速失控。

---

### 问题 3：Message Schema 也过薄，无法支撑后续扩展

当前消息定义偏轻，仅支持少量类型和简化 payload。

问题在于消息至少需要明确：

* 属于哪个 task
* 是否响应某条前序消息
* 本次消息是控制命令还是结果事件
* 本次消息的摘要
* 当前状态变更

否则 inbox 最终会退化成一堆 JSON 文本通知，无法成为稳定的协作接口。

---

### 问题 4：并发一致性要求与技术约束不闭合

文档要求：

* `claim_task()` 要原子认领
* `read_inbox()` 要独占读取并清空
* 文件写入要 write-through-rename
* 同时禁止引入数据库、Redis、锁机制库 

这里的矛盾很明显：

`os.replace()` 只能解决单文件落盘原子性，不能解决“扫描任务 -> 判断可领 -> 占有任务”这一整段逻辑的竞争问题。
`read_inbox()` 读完清空，也无法天然保证多消费者不丢消息。

**结论：当前方案没有真正解决并发一致性，只是提出了目标。**

---

### 问题 5：Agent 生命周期设计偏重，不适合作为首版默认实现

当前 Agent Core 使用 `work_loop + idle_loop` 双循环模式，并带有 sleep polling、identity re-injection、上下文恢复等机制。

这套机制适合长期驻留型 Agent，不适合作为首版最小实现。
对 AI coder 来说，这会显著增加代码量和状态复杂度，导致：

* 首版测试难度升高
* 主流程不易验证
* 出现大量与业务目标无关的循环控制代码

---

### 问题 6：Memory & Compression 模块出现过早，且定位偏核心

当前文档将记忆压缩作为四大核心模块之一，并设计了：

* `microcompact`
* `auto_compact`
* `read_artifact` 

这对长期开放式对话系统有意义，但对当前阶段的多 Agent 协作框架不是核心能力。
更严重的问题是，它容易诱导 AI coder 把“状态正确性”建立在“摘要正确性”上，而不是建立在结构化状态和产物可回读上。

---

### 问题 7：Recorder 角色未真正建模

目标中明确存在“记录”角色，但文档中只有 `event_log.jsonl`，没有 Recorder 的清晰职责、触发条件、输入输出，也未出现在时序图中。

这会导致两种常见后果：

* Recorder 最终被系统日志替代，角色名存实亡
* 记录行为散落在各模块中，审计语义不统一

---

### 问题 8：TDD 方向正确，但还不够“硬约束”

当前文档已经补了 AI coder 的测试驱动章节，包括：

* 分阶段 DoD
* pytest 脚手架
* Mock LLM
* time-lapse 模式 

方向没问题，但仍偏“建议”，不够像“开发契约”。

目前缺少：

* 每阶段必须通过的最小测试列表
* 每阶段禁止实现的内容
* 失败后回退到哪个阶段
* 未通过测试不得进入下一阶段的明确规则

---

### 问题 9：失败语义不统一

文档提到了工具异常、任务失败、状态更新，但没有统一定义：

* tool failure 是什么
* task failure 是什么
* workflow failure 是什么
* 谁负责重试
* 重试如何记录

这会导致各模块对“失败”的理解不同，最终影响恢复和审计。

---

### 问题 10：扩展点没有被正式定义为接口契约

作为通用框架，设计稿应该明确哪些部分未来可以替换。当前虽然提到模块划分，但没有把“可替换边界”正式写成接口。

例如：

* 存储后端是否可从文件切换到 SQLite
* Agent runner 是否可从 Mock 切到真实 LLM
* Tool registry 是否允许本地函数与远端工具并存

这些没有被写成接口契约，后续实现容易被写死。

---

## 三、修订建议

### 建议 1：明确采用“半中心化控制”作为首版主范式

建议把系统控制权统一到一个 Orchestrator。

职责划分建议如下：

* `TaskManager`：只负责任务状态与依赖图管理
* `MessageBus`：只负责消息记录与投递
* `Orchestrator`：负责阶段推进、角色唤起、失败处理
* `Agent`：只负责执行当前任务并返回结果

这样可以避免“Agent 自治拉任务”和“中心调度”混杂。

---

### 建议 2：补强 Task Schema，形成可执行任务单

建议将 Task Node 最少扩展为：

```json
{
  "id": "string",
  "subject": "string",
  "description": "string",
  "role": "planner | executor | reviewer | recorder",
  "status": "pending | in_progress | completed | failed",
  "owner": "string | null",
  "blockedBy": ["string"],
  "input_refs": ["artifacts/x.md"],
  "output_refs": ["artifacts/y.md"],
  "parent_task_id": "string | null",
  "retry_count": 0,
  "failure_reason": "string | null",
  "priority": 100,
  "created_at": "iso8601",
  "updated_at": "iso8601"
}
```

这已经足够支撑首版，又不会太重。

---

### 建议 3：补强 Message Schema，区分 command 与 event

建议消息结构至少补充：

```json
{
  "msg_id": "uuid",
  "timestamp": "iso8601",
  "from": "string",
  "to": "string",
  "kind": "command | event",
  "type": "assign_task | task_done | review_pass | review_reject | retry | shutdown",
  "task_id": "string",
  "reply_to": "uuid | null",
  "summary": "string",
  "payload": {
    "artifact_refs": [],
    "extra": {}
  }
}
```

这样后续 inbox 才不会退化成弱结构通知。

---

### 建议 4：收紧首版并发边界，不假装解决高并发一致性

建议在设计稿中明确写死：

* v1 只支持单个 Orchestrator 进程推进工作流
* 不支持多个执行器同时对同一任务池并发 claim
* inbox 在 v1 中是通知机制，不是强一致消息队列
* 文件原子替换只保障单文件写入完整性，不保障跨文件事务一致性

这样设计是诚实的，也更容易被 AI coder 正确实现。

---

### 建议 5：将 Agent Lifecycle 分为 v1 / v2 两级

建议拆分：

**v1：一次调用型**

* 接到任务
* 组装 prompt
* 调用 LLM / Mock
* 输出结果
* 退出

**v2：常驻轮询型**

* 支持 idle loop
* 支持 inbox 唤醒
* 支持 identity re-injection
* 支持长时间挂起恢复

首版只做 v1，文档中保留 v2 为后续演进方向。

---

### 建议 6：将 Memory/Compression 从核心模块降级为可选能力

建议明确三条：

* 系统状态永不依赖摘要保存
* Task、Artifact、EventLog 才是状态真源
* Memory compression 仅作用于对话消息，不作用于任务事实

这样可保留扩展空间，同时避免状态逻辑被 LLM 摘要污染。

---

### 建议 7：明确 Recorder 的定位

建议二选一：

**方案 A：Recorder 不是独立 Agent**

* 由 Orchestrator 统一记录关键事件
* `event_log.jsonl` 即审计真源

**方案 B：Recorder 是独立角色**

* 接收其他 Agent 的阶段结果
* 输出标准化阶段记录、结论摘要、失败报告

首版更建议采用方案 A，更轻。

---

### 建议 8：把 TDD 改成阶段化 DoD 契约

建议每阶段明确写成：

* 本阶段允许实现什么
* 本阶段禁止实现什么
* 必须通过哪些测试
* 不通过时必须回到哪个阶段修复

例如：

**阶段 1：工作空间与 Schema**

* 允许：workspace 初始化、原子写、Schema 校验
* 禁止：Agent loop、真实模型接入
* 通过标准：pytest 中 `workspace/atomic/schema` 全绿

**阶段 2：任务调度**

* 允许：claim、update、依赖解除
* 禁止：多 Agent 常驻轮询
* 通过标准：DAG 调度与失败重试测试全绿

**阶段 3：Mock Agent 流转**

* 允许：Mock LLM、角色执行、产物落盘、日志追踪
* 禁止：接真实网络 API
* 通过标准：完整闭环仿真测试全绿

---

### 建议 9：补一节统一失败模型

建议补充以下失败语义：

* `tool_failed`：工具执行失败，可在当前任务内重试
* `task_failed`：任务结果未达标，需要 reviewer 或 orchestrator 处理
* `workflow_failed`：关键路径终止，任务整体失败
* `aborted`：人工或系统主动停止

同时规定：

* 失败必须写入 task.failure_reason
* 失败事件必须写入 event_log
* 重试不得覆盖旧记录，只能追加新事件

---

### 建议 10：显式定义扩展接口

建议在文档中加一节“可替换接口”：

* `StorageBackend`
* `AgentRunner`
* `ToolRegistry`

即便首版只有文件系统、本地 Python 工具、Mock LLM，也要先把接口留出来，避免后续重构成本过高。

---

## 四、修订后最小架构

以下是建议采用的“首版最小可落地架构”。

目标不是求全，而是满足：

* 单任务多角色协作
* 可审计
* 可恢复
* 可测试
* 可扩展

---

### 4.1 架构原则

1. 首版采用半中心化控制
2. 状态真源为文件，不依赖内存对话历史
3. 角色执行一次一返回，不做常驻轮询
4. Mock 优先，真实模型后接
5. 先保证流程闭环，再做自治增强

---

### 4.2 模块划分

#### 模块 A：WorkspaceStore

负责：

* 初始化目录
* 原子写入 JSON / JSONL
* 读取 task / artifact / event log

不负责流程控制，不负责 AI 逻辑。

---

#### 模块 B：TaskManager

负责：

* 创建任务
* 获取可执行任务
* 更新状态
* 解除依赖
* 记录失败原因与输出引用

这是流程状态模块，不负责 LLM。

---

#### 模块 C：Orchestrator

负责：

* 推进工作流
* 决定当前调用哪个角色
* 接收角色执行结果
* 写日志
* 处理失败与重试

这是首版控制中枢。

---

#### 模块 D：AgentRunner

负责：

* 根据 role 组装 prompt
* 调用 Mock LLM 或真实 LLM
* 产出结构化结果

首版只做同步单次调用，不做 idle loop。

---

#### 模块 E：ToolSandbox

负责：

* 工具注册
* 参数校验
* 异常捕获
* 工具结果标准化返回

---

#### 模块 F：AuditLog

负责：

* 统一写 `event_log.jsonl`
* 记录阶段推进、产物生成、审核结果、失败原因

首版可由 Orchestrator 内部调用，不一定单独拆进程。

---

### 4.3 最小目录结构

```text
/workspaces/task_{uuid}/
 ├── meta.json
 ├── state.json
 ├── context.json
 ├── event_log.jsonl
 ├── tasks/
 │    ├── task_root.json
 │    ├── task_plan_1.json
 │    └── task_review_1.json
 └── artifacts/
      ├── draft_1.md
      └── review_1.json
```

首版不需要 `.team/inbox/`。
如果未来要支持异步自治 Agent，再补。

---

### 4.4 最小任务流

#### 第一步：Planner

输入原始任务
输出子任务拆解、验收要求、输入输出约束

#### 第二步：Executor

认领执行任务
输出草稿、结果、产物文件引用

#### 第三步：Reviewer

审查执行结果
输出 pass / reject、修改建议

#### 第四步：Orchestrator

根据 Reviewer 结果决定：

* 通过并结束
* 驳回并重派给 Executor
* 达到重试上限后失败结束

#### 第五步：AuditLog

对所有关键事件写审计日志

---

### 4.5 最小状态定义

建议 `state.json` 至少包含：

```json
{
  "workflow_status": "planning | executing | reviewing | completed | failed",
  "current_task_id": "task_review_1",
  "retry_limit": 2,
  "last_error": null
}
```

---

### 4.6 首版不做的内容

为防止 AI coder 过度实现，建议明确首版不做：

* 多进程并发 claim
* 强一致消息队列
* 常驻 Agent 守护循环
* 自动上下文压缩
* 复杂记忆回注
* 数据库后端
* 远端分布式工具调用

---

## 五、最终评审结论

修订前方案的问题，不在于方向错误，而在于：

* 控制范式不统一
* 关键 schema 偏薄
* 并发语义未闭合
* 首版实现范围过大

修订后的建议路线是：

**保留骨架，砍掉首版非必要复杂度，补齐最关键的数据结构和验收契约。**

最终建议如下：

1. 保留文件驱动框架方向
2. 首版采用半中心化 Orchestrator
3. 强化 Task / Message / Failure Schema
4. 将 TDD 升级为 DoD 契约
5. 将常驻 Agent、复杂记忆压缩降为后续能力
