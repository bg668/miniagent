# 一、人审版验收清单

用途：给你或其他评审者做最终验收。
目标：判断 AI coder 的实现是否真正符合设计，而不是“看起来差不多”。

---

## A. 总体判断

### 1. 目标是否对齐

检查点：

* 是否实现的是“单一复杂任务、多角色协作”的框架首版
* 是否仍然是通用多 Agent 雏形，而不是写死成某一个业务脚本
* 是否控制了首版范围，没有擅自扩展成复杂平台

验收标准：

* 是首版协作框架，不是一次性业务脚本
* 有扩展边界，但没有过度实现

常见不通过表现：

* 代码只适配某个固定任务
* 直接做成大而全平台
* 引入大量首版不需要的能力

---

## B. 架构验收

### 2. 控制范式是否正确

检查点：

* 是否由 Orchestrator 统一推进流程
* Agent 是否只执行当前任务，而不是自己扫描全局任务池
* 是否没有混入“半自治 + 半中心化”两套控制逻辑

验收标准：

* Orchestrator 是唯一流程推进者
* Agent 不负责全局调度

不通过表现：

* Agent 自己 claim task
* Agent 自己轮询全局目录
* 流程推进责任不清

---

### 3. 状态真源是否正确

检查点：

* 关键状态是否全部落盘
* 文件是否是系统事实真源
* 是否没有把核心状态偷偷放在内存 messages 或 prompt 中

验收标准：

* `state.json`、`tasks/*.json`、`artifacts/*`、`event_log.jsonl` 都真实存在
* 重启后可以从文件恢复工作流事实

不通过表现：

* 关键状态只存在于对象内存里
* event_log 不完整
* artifact 没有实际落盘

---

### 4. 首版边界是否守住

检查点：

* 是否没有实现首版明确禁止的内容
* 是否没有把复杂能力提前塞进来

验收标准：

* 没有异步 inbox
* 没有多进程并发 claim
* 没有常驻 Agent
* 没有自动记忆压缩
* 没有数据库后端作为默认实现

不通过表现：

* 出现轮询守护进程
* 引入 Redis/SQLite 作为默认核心依赖
* 引入复杂 memory engine

---

## C. 数据模型验收

### 5. Task Schema 是否完整

检查点：

* Task 是否包含设计要求的核心字段
* 任务是否能表达“可执行任务单”，而不是简单待办

重点字段：

* `id`
* `role`
* `status`
* `blockedBy`
* `input_refs`
* `output_refs`
* `retry_count`
* `failure_reason`
* `priority`
* `created_at`
* `updated_at`

验收标准：

* 任务结构完整
* 失败与重试信息可追踪
* 输入输出引用可追踪

不通过表现：

* 只有 subject/description/status
* 没有 failure_reason
* 没有 output_refs

---

### 6. State Schema 是否完整

检查点：

* 是否有全局 workflow 状态
* 是否能看出当前执行到哪个任务、哪个角色
* 失败是否能记录到全局状态

验收标准：

* `workflow_status`
* `current_task_id`
* `current_role`
* `retry_limit`
* `review_round`
* `last_error`

必须可见。

---

### 7. Event Log 是否可复盘

检查点：

* `event_log.jsonl` 是否记录关键流程
* 是否能通过日志还原工作流经过

至少应有：

* workflow start
* task create
* task start
* task complete
* reviewer pass/reject
* task fail
* workflow complete/fail

验收标准：

* 日志足够支撑复盘
* 事件不是无结构文本堆积

不通过表现：

* 只记录少量打印信息
* 没有 reviewer 结论事件
* 没有失败事件

---

## D. 模块职责验收

### 8. WorkspaceStore 是否只做存储

检查点：

* 是否只负责文件读写与原子写
* 是否没有掺入流程控制

验收标准：

* 模块边界单一
* 具备原子写能力

不通过表现：

* 在 store 里做任务调度
* 在 store 里做模型调用

---

### 9. TaskManager 是否只做任务管理

检查点：

* 是否专注任务创建、任务选择、状态更新、依赖解除
* 是否没有掺入模型逻辑

验收标准：

* 任务管理逻辑集中
* 不包含 Agent 调用逻辑

---

### 10. Orchestrator 是否真正承担流程推进职责

检查点：

* 是否完整负责 Planner → Executor → Reviewer 闭环
* 是否根据 reviewer 结果处理 pass/reject/retry/fail

验收标准：

* 流程推进责任清晰
* 失败处理集中在 Orchestrator

不通过表现：

* Reviewer 自己决定重试
* Executor 自己改全局状态
* Orchestrator 只是薄壳转发

---

### 11. AgentRunner 是否被收敛为单次调用

检查点：

* 是否只做角色执行
* 是否没有实现常驻轮询和自治扫描

验收标准：

* 单次调用输入明确
* 单次调用输出结构化

不通过表现：

* 内部有 idle loop
* 内部轮询目录
* 内部负责 claim task

---

### 12. ToolSandbox 是否真正标准化异常

检查点：

* 工具异常是否被转为统一失败结果
* 是否不会直接打崩主流程

验收标准：

* tool error 有统一格式
* 出错后 workflow 还能继续按规则处理

---

## E. 工作流验收

### 13. 标准闭环是否跑通

检查点：

* 是否能跑通 Planner → Executor → Reviewer
* 是否最终能进入 completed 或 failed

验收标准：

* 有完整闭环 demo
* demo 可重复运行
* 输出稳定

---

### 14. Reviewer 驳回逻辑是否成立

检查点：

* reviewer reject 后是否会触发重试
* retry_count 是否增长
* review_round 是否更新

验收标准：

* reject 不只是打印提示
* reject 会改变真实状态与后续流程

---

### 15. 超过上限后是否正确失败

检查点：

* 达到 retry_limit 后是否 workflow failed
* 是否记录 last_error
* 是否写入终止事件

验收标准：

* 失败路径完整
* 没有无限重试

---

## F. 失败模型验收

### 16. 失败语义是否统一

检查点：

* tool_failed / task_failed / workflow_failed 是否有明确区分
* 各模块是否复用统一语义

验收标准：

* 状态定义清楚
* 日志、任务、全局状态使用统一术语

不通过表现：

* 每个模块各自发明失败状态
* failed/error/reject 混用且无边界

---

### 17. 失败是否能追踪

检查点：

* task 失败后是否有 `failure_reason`
* workflow 失败后是否有 `last_error`
* event_log 是否记录失败事件

验收标准：

* 任一失败都能追踪来源和阶段

---

## G. 测试验收

### 18. 是否按阶段开发

检查点：

* 是否能看出阶段化测试
* 是否不是一次性堆完整实现

验收标准：

* workspace/schema 测试独立
* task_manager 测试独立
* orchestrator mock flow 测试独立
* tool_sandbox 测试独立

---

### 19. 测试是否验证关键行为

检查点：

至少要覆盖：

* 初始化工作空间
* 原子写
* 任务选择
* 依赖解除
* reviewer reject + retry
* retry limit fail
* event log 完整链路

验收标准：

* 测试验证关键行为，不只是 smoke test

---

### 20. Mock 闭环是否真实有效

检查点：

* Mock Planner / Executor / Reviewer 是否真的参与流程
* 是否不是硬编码直接写最终结果

验收标准：

* mock 能模拟 pass/reject 两种路径
* demo 有真实阶段推进

---

## H. 最终结论模板

你可以按下面口径做最终结论：

### 通过

* 架构符合 v2 设计
* 首版边界守住
* 闭环完整
* 日志与状态可复盘
* 测试覆盖关键路径

### 有条件通过

* 主流程正确
* 但某些 schema、失败记录、测试覆盖仍需补齐

### 不通过

* 控制范式偏离
* 状态未落盘
* 首版边界失守
* 闭环未跑通
* 测试不能证明实现有效
