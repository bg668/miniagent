**AI Agent 轻量级协同系统：底层抽象与核心模块设计规范 (v1.0 研发参考版)**

本规范作为后续代码编写的全局蓝图，基于“纯文件驱动 (File-backed FSM)”与“黑板模式 (Blackboard)”的原则，对系统的各个模块边界、核心数据 Schema 及交互接口进行严格界定。

---

### 一、 核心数据模型与持久化结构 (Data Schemas & Storage)

所有协同状态必须以文本或 JSON 形式落盘，内存仅作为计算与控制流的临时载体。

#### 1. 工作空间目录拓扑 (Workspace Topology)
```text
/workspaces/task_{uuid}/
 ├── meta.json         # 只读：任务初始参数与全局元数据
 ├── state.json        # 高频写：状态机游标 (PLANNING, EXECUTING...)
 ├── context.json      # 共享：多 Agent 推导出的提炼信息 (KV 结构)
 ├── event_log.jsonl   # 追加：审计流与历史动作追踪
 ├── tasks/            # 依赖图：拆解出的子任务文件库
 │    ├── task_1.json
 │    └── task_2.json
 ├── .team/inbox/      # 消息总线：各 Agent 的本地收件箱
 │    ├── Dispatcher.jsonl
 │    └── Executor.jsonl
 └── artifacts/        # 过程产物：代码、长文本报告、大 JSON 响应
      └── step_1_draft.md
```

#### 2. 关键实体 Schema 界定

**A. 子任务实体 (Task Node)** - 存储于 `tasks/task_{id}.json`
基于有向无环图 (DAG) 理念设计，通过 `blockedBy` 实现依赖阻塞。
```json
{
  "id": "int",
  "subject": "string",
  "description": "string",
  "status": "enum: pending | in_progress | completed | failed",
  "owner": "string | null", 
  "blockedBy": ["int"] // 依赖的 Task ID 列表。为空时才可被 claim
}
```

**B. 消息总线实体 (Event Message)** - 追加于 `.team/inbox/{agent}.jsonl`
仅传递控制指令与文件指针，**严禁传递超过 1KB 的数据体**。
```json
{
  "msg_id": "uuid",
  "timestamp": "float",
  "from": "string (Agent Name)",
  "type": "enum: assign_task | file_ready | reject | shutdown",
  "payload": {
    "instruction": "string",
    "artifact_refs": ["artifacts/xxx.md"] // 指向具体产物文件的相对路径
  }
}
```

### 二、 组成模块功能界定与对外接口 (Modules & Interfaces)

整个系统划分为四大核心模块：消息总线与调度器、智能体生命周期内核、记忆与防溢出引擎、工具沙箱。

#### 模块 1: 消息总线与任务调度 (MessageBus & TaskManager)
* **功能界定**：纯粹的 I/O 模块。负责多 Agent 之间的 JSONL 消息投递、状态文件 (`state.json`, `context.json`) 的原子化读写，以及基于 `blockedBy` 的任务图维护。自身**不包含任何 AI 逻辑**。
* **核心接口**：
    * `send_message(sender: str, to: str, msg_type: str, payload: dict) -> None`: 追加单行 JSON 至目标 Inbox。
    * `read_inbox(agent_name: str) -> list[dict]`: 独占式读取并**清空**该 Agent 的 Inbox 文件内容。
    * `claim_task(agent_name: str) -> dict | None`: 扫描 `tasks/` 目录，返回第一个 `status=pending` 且 `blockedBy=[]` 的任务，并原子化更新其 owner。
    * `update_task(task_id: int, status: str) -> None`: 当状态更新为 `completed` 时，触发级联扫描，将自身 ID 从其他任务的 `blockedBy` 列表中抹除。

#### 模块 2: 智能体内核 (Agent Core Lifecycle)
* **功能界定**：承载 LLM 实例与角色设定的执行单元。采用双循环状态机 (`Work Phase` & `Idle Phase`)，支持长时间挂起与事件唤醒。
* **数据载体**：维护自身的局部短期记忆 `messages: list[dict]`（不超过 Token 阈值）。
* **内部流转逻辑 (伪代码)**：
    ```python
    class BaseAgent:
        def _work_loop(self):
            # 1. 消费 inbox 与 context.json 组装 prompt
            # 2. 调用 LLM 
            # 3. 拦截 tool_calls 并执行
            # 4. 若 LLM 决定挂起，或无后续 tool_call，进入 idle
            pass
            
        def _idle_loop(self):
            # 1. sleep(POLL_INTERVAL)
            # 2. 检查 inbox 是否有新消息
            # 3. 检查 TaskManager 是否有可认领的 pending 任务
            # 4. 若发现工作，重新注入 <identity> 身份信息，切换回 work_loop
            pass
    ```

#### 模块 3: 记忆与细节保真引擎 (Memory & Compression Engine)
* **功能界定**：针对复杂任务产生的海量 Token 历史，实施无损/有损结合的动态折叠，防止 LLM Token 溢出与幻觉。
* **核心接口**：
    * `microcompact(messages: list) -> None`: (无损) 扫描近期 `tool_result`，当工具返回结果长于 2 轮对话时，将其正文替换为 `[cleared]` 释放空间，迫使 LLM 依赖更紧凑的总结。
    * `auto_compact(messages: list) -> list`: (有损压缩) 触发阈值后，将所有历史转储至 `TRANSCRIPT_DIR`，单独调用一次 LLM 归纳核心 Summary。将当前 `messages` 暴力替换为 `[{"role": "user", "content": "Summary + Transcript File Path"}]`。
    * `read_artifact(path: str) -> str`: (细节保真工具) 当 Agent 在压缩后的 Context 中丢失了早期数据结构细节时，强制要求其调用此 Tool 回溯读取 `artifacts/` 下的原始文件。

#### 模块 4: 工具沙箱与契约校验 (Tool Sandbox & Validation)
* **功能界定**：将 Python 原生函数转化为 LLM 友好的 JSON Schema。捕获所有底层运行崩溃，实现错误反射 (Self-Correction)。强制结构化产出。
* **对外接口**：
    * `@tool(name, desc, schema)` 装饰器：自动注册函数至可用能力池。
    * `execute_tool(tool_name: str, kwargs: dict) -> str`: 拦截器。
* **防爆边界**：所有 Tool 执行必须包覆在 `try-except` 中。如果工具崩溃，返回截断后的 Traceback（例如 `{"error": "KeyError 'url' at line 45", "status": "failed"}`），交由 LLM 消化并重试，**绝对禁止将异常抛给 Agent 主线程导致进程死亡**。

### 三、 典型复杂任务流转时序 (Sequence Diagram)

     │                                     │                                  │                              │

     ├─ write -> tasks/task_1.json ───────►│                                  │                              │
     ├─ write -> tasks/task_2.json ───────►│                                  │                              │
     │                                     │ 2. Idle Polling 扫描无阻塞任务      │                              │
     │                                     │◄────────── claim_task() ─────────┤                              │
     │                                     │                                  │ 3. 执行任务 (Work Phase)       │
     │                                     │                                  ├─ call_llm()                  │
     │                                     │ 4. 产物落盘                        ├─ web_scrape()                │
     │                                     │◄─ write -> artifacts/result.md ──┤                              │
     │                                     │                                  │ 5. 通知审核                   │
     │                                     │◄─ append -> inbox/Reviewer.jsonl ┤                              │

### 四、 补充规范：原子级防损与重试

为了保证断点续传的 100% 可靠，必须在底层 I/O 封装时强制遵循以下约定：

1.  **写时复制 (Write-Through-Rename)**：针对 `state.json` 或 `task_x.json` 等状态文件的覆盖写操作，严禁直接 `open(file, 'w')`。必须：
    * 写入 `state.json.tmp`
    * 执行 `os.fsync()` 确保刷入磁盘
    * 执行 `os.replace('state.json.tmp', 'state.json')` 实现原子级替换。
2.  **API 抖动退避**：在 Agent Core 中，LLM API 请求必须包覆指数退避 (Exponential Backoff, 库推荐：`tenacity`)，区分 `429 Too Many Requests`（长退避）与 `5xx Server Error`（短重试）。
3.  **身份重注 (Identity Re-injection)**：当 Agent 从长时间的 `Idle` 状态被唤醒，且恰逢 Context 触发了 `auto_compact` 时，其内存中的 System Prompt 意图会被稀释。此时引擎必须强制在 `messages` 头部重新推入一条：`<identity>You are Executor, your current goal is...</identity>`，保持智能体的认知锚点不偏离。

### 五、 AI 开发验收标准与测试驱动沙盒 (AI Coder Evaluation Stack)

本章节专为接手此项目的 AI 辅助开发工具（AI Coder）设计。AI 在开发本系统时，必须遵循“测试驱动开发 (TDD)”模式。系统的可观测性由自动化的测试沙盒与断言脚本提供，以确保 AI 输出的代码严格符合文件系统状态机设计。

#### 5.1 AI 阶段性验收标准 (Definition of Done)

AI Coder 必须分三个阶段进行开发，每个阶段完成后必须通过对应的可观测性测试：

* **阶段 1：底层 I/O 与 Workspace 模块开发**
    * **验收标准**：能在临时目录正确生成包含 `meta.json`、`state.json`、`context.json` 等规范结构的工作空间。
    * **断言关键**：验证“写时复制 (Write-Through-Rename)”的原子性；多线程并发写入同一个状态文件时不抛出文件损坏异常。
* **阶段 2：TaskManager 与状态机调度**
    * **验收标准**：能够正确解析 `tasks/` 目录下的 `blockedBy` 依赖图；`claim_task()` 能正确返回拓扑排序中最优先的任务；状态更新能正确抹除依赖。
* **阶段 3：多 Agent 流转沙盘模拟 (Mocked LLM)**
    * **验收标准**：用伪造的 Mock Agent（不实际调用 LLM API，仅读写本地文件）跑通一个完整的 `PLANNING -> EXECUTING -> REVIEWING` 闭环。

#### 5.2 核心测试脚手架 (Test Harness Schema)

为了让 AI Coder 能够“观测”自己的开发结果，系统必须预先定义一套基于 `pytest` 的测试脚手架。AI 必须先写测试，再写业务代码。

**材料补充要求：需向 AI 规定 `conftest.py` 的行为**
AI 需要在一个隔离的临时目录（`tempfile.TemporaryDirectory`）中运行所有测试，绝对禁止在真实的系统目录中读写文件。

```python
# 示例：要求 AI Coder 必须实现的测试断言结构
def test_workspace_atomic_write(temp_workspace):
    """
    验证 AI Coder 实现的文件写入是否具备原子性，防崩溃。
    可观测性指标：断言 .tmp 文件是否存在，最终文件内容是否完整。
    """
    workspace = TaskWorkspace(temp_workspace.id)
    workspace.update_state("EXECUTING")
    
    assert os.path.exists(temp_workspace.path / "state.json")
    assert not os.path.exists(temp_workspace.path / "state.json.tmp")
    assert json.loads((temp_workspace.path / "state.json").read_text())["status"] == "EXECUTING"
```

#### 5.3 无大模型沙盘驱动器 (Mock Simulator)

为了验证 AI Coder 写出的 Agent 协同逻辑是否正确，**绝不能让 AI 边测边调用真实的 OpenAI/Anthropic API**（既慢又不可控）。需要向 AI Coder 补充一个“模拟沙盘”模块需求。

**规定 AI Coder 实现以下 Mock 机制：**
* **Mock LLM Client**：强制 AI 拦截掉所有的网络请求。根据输入的 Prompt 关键词，返回预设的硬编码 JSON 结果。
* **Time-lapse 模式**：将 Agent 的长轮询 (Idle Polling) `sleep(5)` 缩短为 `sleep(0.01)`，以便在 1 秒内跑完一个极其复杂的、原本需要数小时的多 Agent 协作流程。

**可观测性输出示例 (Simulator Stdout)**：
要求 AI 编写的沙盘启动脚本必须输出如下标准化的流转日志，供人类开发者或上级 AI 审查：
```text
[0.01s] [Dispatcher] Claimed task_1. Status -> EXECUTING
[0.02s] [Dispatcher] Wrote sub-tasks to tasks/task_1_1.json
[0.03s] [Executor] Claimed task_1_1 (blockedBy=[]). Status -> RUNNING
[0.05s] [Executor] Artifact generated: artifacts/draft.md. Inbox msg sent to Reviewer.
[0.06s] [Reviewer] Inbox drained. Artifact read. Status -> COMPLETED
```

#### 5.4 针对 AI Coder 的“强制纠偏”指令约定 (Prompting Contracts)


### 六、开发约束

1.  **架构不可变原则**：明确告知 AI “本《设计方案》的架构（纯文件系统状态机、无数据库）已锁定。如果你在开发时遇到并发或性能困难，只能通过优化本地 I/O 解决，**绝对禁止引入 SQLite、Redis 或任何锁机制库**”。
2.  **红绿重构协议**：规定 AI 在修改任何核心模块（如 `MessageBus`）后，第一步必须是静默运行 `pytest tests/`，并将错误堆栈提取出来进行自我修复。只有当控制台全部报绿（Passed）时，才允许向人类报告进度。
3.  **单点验证优先**：禁止 AI 一次性生成全部几千行代码。要求其“先生成数据模型与目录初始化代码，提供一个测试脚本运行确认无误后，再继续写状态机流转”。
