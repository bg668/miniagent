# 基于 TypeScript 参考实现的 Agent Python 重写概要设计

## 一、文档目的

本文档用于指导在 `agent_sdk` 目录下重写一个 Python 版 Agent Runtime。该 Runtime 的行为语义需要严格对齐 `refs` 目录中的 TypeScript 参考实现，但本次实现范围只覆盖当前明确需要的能力，不追求一次性实现参考实现的全部平台特性。

本次设计强调两点。第一，必须保持参考实现中与运行时行为相关的核心语义不变，例如消息何时进入上下文、事件何时发出、工具调用何时执行、一次运行何时结束、出错和中止时如何收敛。第二，只实现当前确有需求的能力，避免引入代理流、复杂 provider 适配或额外平台参数，控制实现边界。

## 二、实现范围

本次 Python 版本只需实现以下三类能力。

第一，Agent LOOP 的运行状态需要外部可感知，且 LOOP 可以在特定节点被取消；在取消后，允许通过添加新的用户提示词继续执行。这一能力对应参考实现中的活动运行状态、事件流、steering 队列、follow-up 队列、`abort()`、`waitForIdle()`、`prompt()` 和 `continue()` 的组合语义。

第二，不修改 Agent LOOP 本身，即可在应用层扩展工具、加载技能、管理上下文、修改系统提示词、创建计划任务等功能。这里的要求不是在 LOOP 内直接实现这些业务功能，而是要求 LOOP 提供足够稳定的扩展点和边界，使这些功能作为上层能力接入。

第三，只适配 OpenAI 风格的 `chat.completions` 接口，包括 stream 和非 stream 两种调用模式。也就是说，本次不实现 `proxy.ts` 对应的代理协议，不实现多 provider 兼容，只实现一个面向 OpenAI 风格消息协议的模型适配层。

## 三、设计目标

本次设计的核心目标，是建立一个可被外部应用调用的 Python `agent_sdk`。这里的 `agent_sdk` 不是应用宿主工程，而是一个对外提供可扩展 Agent LOOP 能力的 SDK 包。

这个 SDK 负责会话推进本身，包括消息状态、turn 边界、模型调用、工具调用、取消、中止和继续执行。它应当足够稳定，不应因为外部应用增加技能管理器、任务规划器或上下文注入器就被迫改动主循环。

外部应用负责扩展行为，包括新增工具、通过预处理注入技能说明、动态裁剪或补充上下文、在特定 turn 结束后追加用户提示、根据外部状态修改系统提示词等。外部应用对 SDK 的接入方式应当通过工具列表、hook、队列、上下文转换函数和配置对象完成，而不是直接改写 LOOP 逻辑。

## 四、严格对齐的语义基线

虽然本次实现范围有所收缩，但以下语义仍然必须严格对齐参考实现。

Agent 是一个单活动运行对象。同一时刻只允许一个运行中的 prompt 或 continue 任务。新的运行请求不能隐式打断当前任务，也不能自动排队覆盖当前任务。

会话推进采用双层循环。内层循环负责处理当前 pending 消息、请求 assistant 回复、执行工具并继续下一轮；外层循环负责在内层看似结束时检查 follow-up 消息，有则重新进入内层，无则真正结束。

模型回复需要支持流式 partial message 收敛为 final message 的语义。也就是说，流式过程中上下文中的最后一条 assistant 消息是持续更新的“当前快照”，在流结束时被最终消息覆盖，而不是每个增量片段都变成独立消息。

工具调用需要保留“准备、校验、前置 hook、执行、后置 hook、结果落地”的完整链路。顺序模式和并行模式都要保持与参考实现一致的行为，尤其是并行模式中的“准备阶段按顺序，执行阶段并发，结果按原工具调用顺序落地”这一点不能改变。

错误和中止应优先编码为协议内可消费结果，而不是让运行时直接崩溃。assistant 失败或中止应落为带 `stop_reason=error` 或 `stop_reason=aborted` 的消息；工具失败应落为 `is_error=true` 的 tool result。

## 五、范围内必须实现的能力

### 5.1 LOOP 状态外部可感知

外部需要看到当前 Agent 是否正在运行、当前是否存在流式中的 assistant 消息、当前有哪些工具调用正在执行、最近一次失败的错误信息是什么，以及本次运行已经发出了哪些生命周期事件。

因此，Python 版本必须提供公开状态对象和事件订阅接口。状态对象至少需要包含：完整消息列表、`is_streaming`、`streaming_message`、`pending_tool_calls`、`error_message`、当前模型、当前系统提示词和当前工具集合。事件订阅接口至少需要支持：`agent_start`、`agent_end`、`turn_start`、`turn_end`、`message_start`、`message_update`、`message_end`、`tool_execution_start`、`tool_execution_update`、`tool_execution_end`。

外部“感知 LOOP 状态”的方式，应该优先通过订阅事件和读取状态快照完成，而不是依赖内部私有字段或打印日志。

### 5.2 在特定节点取消执行

本次实现不要求任意指令级抢占，也不建议做“随时硬中断”的不稳定设计。严格对齐参考实现时，取消应依赖统一的取消信号，并在模型流读取、工具执行和事件监听等待这些自然边界上生效。

换句话说，取消点应至少包括以下位置：模型流下一次读取前、工具执行内部显式检查取消信号时、hook 执行过程中、以及等待事件监听器完成的阶段。只要在这些边界取消成功，运行就应收敛为一次 `aborted` 结束，而不是残留半更新状态。

### 5.3 取消后添加新提示词继续执行

参考实现中并不存在“中止后恢复同一个 call stack”的设计，而是通过消息队列和新的运行入口实现继续推进。因此，Python 版本也应遵循同样的思想：取消后保留已收敛的 transcript，然后允许上层通过 `prompt()` 新增用户消息，或通过 steering / follow-up 队列插入新消息，再次启动新的运行。

这意味着“继续执行”的本质不是恢复某个被暂停的底层生成器，而是以保留下来的上下文为基础，发起下一次受控运行。

### 5.4 不改 LOOP 即可扩展应用层能力

本次是架构上的重点。SDK 内核必须预留以下扩展面：

工具扩展通过 `tools` 列表注入，不改 LOOP。

技能加载通过系统提示词拼装、上下文注入或 steering 消息注入完成，不改 LOOP。

上下文管理通过 `transform_context` 扩展点完成，例如压缩历史、注入摘要、补充外部状态，不改 LOOP。

系统提示词修改通过 Agent 状态或运行前配置调整完成，不改 LOOP。

计划任务创建通过应用层工具、follow-up 消息或外层调度器配合实现，不改 LOOP。

这些能力都必须依赖统一接口接入，而不能要求外部应用 fork 一份 LOOP 逻辑。

### 5.5 只适配 OpenAI 风格 `chat.completions`

本次只实现 OpenAI 风格的消息协议和工具调用协议。模型适配层只需要支持：

非流式 `chat.completions.create(..., stream=False)`。

流式 `chat.completions.create(..., stream=True)`。

工具调用使用 OpenAI 风格 tool call 结构。

因此，本次不实现代理协议重建，不实现多 provider 差异映射，也不实现单独的 transport 抽象层。运行时内部仍可以保留流式接口抽象，但其唯一适配实现只需要覆盖 OpenAI 风格输入输出。

## 六、总体架构

总体上建议将实现划分为两部分。

第一部分是 `agent_sdk` 自身，也就是对外暴露的 Runtime Kernel / Agent SDK。它负责消息和事件模型、状态管理、主循环、流式回复处理、工具执行、取消与收尾。这个层次必须尽量通用和稳定，不能混入技能系统、任务编排策略或具体业务逻辑。

第二部分是 `agent_sdk` 之外的外部应用层。外部应用负责工具注册、技能装载、上下文压缩、系统提示词拼装、计划任务策略以及外部状态同步，并通过 SDK 暴露的配置和扩展点与 Runtime Kernel 交互。

二者之间的边界需要非常清晰：`agent_sdk` 不知道“技能”是什么，也不知道“计划任务”是什么；它只知道工具、消息、上下文变换、系统提示词和待处理消息队列。

## 七、核心模块边界

代码根目录为 `agent_sdk`，建议按 SDK 包方式组织。

`agent_sdk/runtime/models.py` 用于定义运行时内部消息模型、工具模型、上下文模型和公共类型。这里应包含 user、assistant、tool result、内容块、tool call、agent state 和 tool result 的基础结构。

`agent_sdk/runtime/events.py` 用于定义事件模型和监听器类型，保证状态归约和外部订阅使用统一事件结构。

`agent_sdk/runtime/state.py` 用于定义公开状态和内部可变状态，重点承载 `is_streaming`、`streaming_message`、`pending_tool_calls`、`error_message` 和 transcript。

`agent_sdk/runtime/queues.py` 用于实现 steering 队列和 follow-up 队列。

`agent_sdk/runtime/agent.py` 用于实现对外 Agent 对象，负责公开 API、状态管理、事件分发、取消、等待 idle、重置、队列操作。

`agent_sdk/runtime/loop.py` 用于实现 `run_agent_loop()`、`run_agent_loop_continue()` 和双层主循环。

`agent_sdk/runtime/stream_handler.py` 用于把 OpenAI 风格 stream 或非 stream 响应统一转成 assistant final message，并在流式时维护 partial message。

`agent_sdk/runtime/tool_executor.py` 用于实现工具调用的准备、执行、并行控制、hook 和 tool result 落地。

`agent_sdk/adapters/openai_chatcompletions.py` 用于封装 OpenAI 风格 `chat.completions` 调用，并产出运行时需要的统一事件流接口。

外部应用层能力不应放进 `agent_sdk` 包内，它们应位于调用该 SDK 的上层工程中，并通过 SDK 暴露的接口接入。

## 八、主循环设计

主循环仍然采用与参考实现一致的双层循环。

内层循环的职责，是不断消费 pending 消息并推进一个个 turn。一个 turn 的基本结构是：必要时发出 `turn_start`，将 pending 消息按 `message_start` 和 `message_end` 的顺序注入上下文，调用模型获取一条 assistant 回复，若 assistant 包含工具调用则执行工具，并将工具结果加入上下文，最后发出 `turn_end`。

外层循环只负责一个判断：当内层循环结束后，系统是否真的应该停下。如果应用层通过 follow-up 队列提供了新的消息，则这些消息会变成下一轮 pending 消息，Agent 不结束，继续进入内层循环。如果没有 follow-up，整次运行才发出 `agent_end`。

这套结构必须保留，因为它直接决定“中途插入新提示词”和“本轮结束后再追加提示词”这两种能力的时机差异。

## 九、流式与非流式统一设计

虽然本次只适配 OpenAI 风格 `chat.completions`，但仍然应在运行时内部统一抽象为“消息事件流 + 最终结果”的协议。

对于流式调用，适配层应把 OpenAI chunk 增量转换为 `start`、`text_delta`、`toolcall_delta`、`done` 这类运行时事件，并由 `stream_handler` 维护 partial assistant message。

对于非流式调用，适配层可以退化为一个只返回 final assistant message 的单次结果流。也就是说，在运行时内部，即使底层是非流式，也最好仍通过统一接口返回一个可收敛的 final message，这样 LOOP 不需要为 stream 和 non-stream 写两套逻辑。

## 十、工具执行设计

工具执行必须保留参考实现的流水线语义。

首先从 assistant message 中提取 tool call。然后针对每个 tool call 依次做：工具查找、参数预处理、参数校验、`before_tool_call`、工具执行、`after_tool_call`、生成 tool result 消息。

如果找不到工具、参数校验失败，或 `before_tool_call` 明确阻断，则本次工具调用直接转成错误 tool result，而不是抛异常中断整个 LOOP。

如果工具执行抛错，也必须转换为错误 tool result。

并行模式下，只有工具执行阶段允许并发；前置准备阶段和最终结果落地阶段仍按原 tool call 顺序执行，以保证事件顺序和 transcript 顺序稳定。

## 十一、扩展点设计

为了满足“不改 LOOP 即可扩展外部应用功能”的要求，运行时需要明确提供以下扩展点。

`tools`：应用层通过传入工具列表扩展能力。

`system_prompt`：应用层可以在运行前设置或更新系统提示词。

`transform_context(messages, cancel_token)`：应用层可以在每轮模型调用前裁剪、压缩或注入上下文。

`convert_to_llm(messages)`：应用层可以控制内部消息到 OpenAI 风格消息的映射方式。

`before_tool_call(context, cancel_token)`：应用层可以在工具执行前做鉴权、拦截或记录。

`after_tool_call(context, cancel_token)`：应用层可以修改工具结果内容和错误标记。

`steer(message)`：应用层可以在当前运行过程中插入新的用户提示词或控制消息。

`follow_up(message)`：应用层可以在当前运行自然结束后追加新的消息，继续推进任务。

借助这些扩展点，技能系统、上下文管理器、系统提示词拼装器、计划任务创建器都可以作为外部应用组件实现，不需要入侵 LOOP 本身。

## 十二、取消与继续执行设计

取消能力应设计成 Runtime Kernel 的标准能力，而不是应用层私有逻辑。

Agent 对外提供 `abort()`。一旦调用，当前运行持有的取消信号被置位，模型流读取、工具执行和 hook 都应尽快在下一个检查点收敛。收敛结果不是丢失消息，而是产生一次带 `aborted` 语义的 assistant final message 或由外层失败包装逻辑补一条错误 assistant 消息。

取消之后，Agent 仍然保留当前 transcript 和 SDK 状态。外部应用可以选择三种继续方式：直接再调用 `prompt()` 加入新用户提示词；在下一轮前调用 `steer()` 插入控制消息；或在本轮自然停下后通过 `follow_up()` 追加后续提示。这个继续模型与参考实现一致，强调“基于已落地上下文重新启动下一次运行”，而不是恢复某个未完成的底层生成器。

## 十三、最小实现清单

为了满足当前需求，第一阶段必须完成以下能力。

实现可观察的 Agent 状态对象和事件订阅机制。

实现 `prompt()`、`continue_()`、`abort()`、`wait_for_idle()`、`steer()`、`follow_up()`。

实现与参考实现语义一致的双层 LOOP。

实现 OpenAI 风格 `chat.completions` 的 stream 和非 stream 适配。

实现 partial assistant message 到 final assistant message 的收敛逻辑。

实现顺序和并行两种工具执行模式。

实现 `transform_context`、`before_tool_call`、`after_tool_call` 这些应用层扩展点。

实现错误与中止的协议化收尾。

## 十四、本次明确不做的内容

本次不实现 `refs/proxy.ts` 对应的代理流式协议。

本次不实现多 provider 抽象与适配。

本次不实现复杂的计划任务引擎本体，只保留让外部应用可以创建计划任务的扩展接口。

本次不实现技能系统本体，只保留让外部应用装载技能并通过系统提示词、上下文和工具接入的能力。

本次不实现长期记忆、数据库持久化和自动压缩策略本体，只保留上下文管理扩展点。

## 十五、验收标准

验收时不应只验证“能返回答案”，而应验证以下语义是否与参考实现一致。

一是外部是否能够稳定观察 LOOP 状态，包括运行中状态、流式消息、工具执行状态和错误状态。

二是 `abort()` 是否能够在模型流和工具执行的边界上稳定生效，并正确收敛为一次已结束运行。

三是在取消或结束后，新增用户提示词是否可以基于现有 transcript 再次推进任务。

四是外部应用新增工具、技能装载、上下文管理、系统提示词修改、计划任务创建，是否都能在不修改 LOOP 的前提下接入。

五是 stream 和非 stream 两种 OpenAI 风格接口是否都能被统一接入主循环。

六是工具并行模式下，结果顺序、事件顺序和 transcript 顺序是否仍与原 tool call 顺序一致。

只有以上行为都成立，才可以认为本次 Python 重写在当前需求范围内严格对齐了 `refs` 中的 TypeScript 语义。
