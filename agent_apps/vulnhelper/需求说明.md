这是基于标准 Agent SDK 工具调用（Tool Calling）机制，重新校准后的四阶段系统设计需求规格。此版本明确了解耦逻辑，确保在利用 LLM 原生工具生成能力的同时，维持对核心业务流程的绝对控制。

### 阶段一：意图收敛与模拟规划 (Intent Convergence & Simulated Planning)

* **需求描述：** 接收用户非结构化输入，进行语义解析。此时不触发真实的数据检索，而是进行“预执行推演”，提取关键查询实体并向用户明确将要执行的检索策略，等待授权指令。
* **需求应答方式：** 文本追问或策略确认话术（如：“提取到查询目标为 X，版本 Y，是否为您执行检索？”）。Agent 进入 `WAITING_FOR_CONFIRMATION` 状态。
* **职责边界：**
    * **LLM (大脑)：** 担任**翻译官与规划师**。负责 NLU 解析，提取出 `product`, `version`, `risk_level` 等潜在参数，并输出一段通顺的确认话术。**此时受 Prompt 严格约束，不触发任何 Tool Call。**
    * **Human (决策者)：** 担任**审核与授权者**。核对 LLM 提取的实体是否偏离预期，补充缺失的上下文参数，或输入“确认/同意”放行。
    * **Program (执行与排版器)：** 担任**状态流转控制器**。记录 LLM 解析出的临时参数（`planned_args`），拦截底层流转，将确认话术抛给前端展示，并挂起当前会话，监听用户的授权动作。

---

### 阶段二：受控工具调用与确定性执行 (Controlled Tool Calling & Deterministic Execution)

* **需求描述：** 获得授权后，正式生成查询参数并执行底层漏洞库查询。此阶段将自然语言意图转化为硬核代码逻辑，完成实体归一化、版本区间覆盖计算，并实现数据的内存级驻留。
* **需求应答方式：** 后台静默执行。如果遇到致命错误（如数据库连接断开），输出标准的系统级报错提示，否则不向前端抛出任何中间思考过程。
* **职责边界：**
    * **LLM (大脑)：** 担任**参数构造器**。接收到用户的“确认”指令和阶段一的 `planned_args` 后，触发原生的 Tool Calling 能力，严格按照已确认的参数结构，生成标准的函数调用请求。**受系统级 Prompt 强制约束，严禁在此阶段发散、篡改或自行添加未确认的参数。**
    * **Human (决策者)：** 等待执行结果。
    * **Program (执行与排版器)：** 担任**执行核心与记忆中枢**。
        1. 接收 LLM 抛出的 Tool Call 请求，校验参数合法性。
        2. 触发本地 Python 工具，执行版本重叠推演与实体查询逻辑。
        3. 接收返回的结构化 JSON 数据集，并将其写入当前 Agent 会话的 `Session Cache` 中，阻断全量数据直接暴露给 LLM。

---

### 阶段三：混合组装与专家研判 (Hybrid Assembly & Expert Analysis)

* **需求描述：** 将阶段二获取的硬性统计数据与 LLM 动态生成的软性分析文本相结合，渲染出格式稳定、无废话的高质量安全研判报告。
* **需求应答方式：** 渲染完成的 Markdown 结构化卡片（固定分为：排查对象、检索统计、专家研判与修复推演）。
* **职责边界：**
    * **LLM (大脑)：** 担任**安全咨询专家**。读取 Program 截取的关键统计变量（如：当前版本、全局最低修复版本、同版本线最高修复版本），不涉及具体 CVE 列表，用专业口吻生成针对性的升级或打补丁建议文本。
    * **Human (决策者)：** 阅读结构化总结报告，获取最终的安全运营决策依据。
    * **Program (执行与排版器)：** 担任**排版渲染引擎**。
        1. 将阶段二产出的核心数据指标作为上下文推送给 LLM 换取分析文本。
        2. 唤醒模板引擎（如 Jinja2），将本地数据的绝对值与 LLM 输出的研判文本注入预设的 Markdown 骨架中。
        3. 将最终组装完毕的文本推流至前端。

---

### 阶段四（可选）：内存下钻与局部过滤 (In-Memory Drill-down & Local Filtering)

* **需求描述：** 响应用户基于报告提出的明细查看需求。避免重复发包查询数据库，直接在内存层完成列表数据的清洗与展示。
* **需求应答方式：** 紧凑的 Markdown 数据表格。
* **职责边界：**
    * **LLM (大脑)：** 担任**指令编译器**。将自然语言（如：“只看高危的”）转化为 JSON 结构的过滤参数（如 `{"filter_key": "risk", "value": "high"}`）。
    * **Human (决策者)：** 发出进一步过滤或下钻查看明细的指令。
    * **Program (执行与排版器)：** 担任**数据处理与展示层**。
        1. 提取 LLM 生成的过滤参数。
        2. 访问阶段二存入 `Session Cache` 中的完整数据集。
        3. 通过 Python 原生代码执行 List/Dict 的内存级过滤。
        4. 按照预设的表格 Schema 输出最终结果。
