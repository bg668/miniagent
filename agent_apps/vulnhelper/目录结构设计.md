# VulnHelper 目录结构设计

## 一、设计目标

本文档用于定义 `agent_apps/vulnhelper/` 的目标目录结构、每层职责以及文件之间的依赖方向，作为后续实现和代码评审时的结构约束。

目录设计遵循以下原则：

- 业务编排、领域逻辑、基础设施与展示层分离
- `agentsdk` 在当前目录下 vendoring 一份副本，并由 `vulnhelper` 本地引用
- 本地漏洞数据库固定放在 `data/`，不与缓存或测试数据混放
- 模板、提示词、测试样例都以文件形式落地，避免硬编码进 Python 源码

## 三、目录职责

### 3.1 根目录文件

- `README.md`
  对外说明运行方式、依赖、环境变量和快速开始。
- `app.py`
  作为应用入口，适合 CLI 或最小 HTTP 启动。
- `bootstrap.py`
  负责对象装配，把 Repository、Cache、Renderer、Agent 和 Orchestrator 组装起来。
- `config.py`
  负责读取模型配置、数据库路径、默认提示词路径和运行参数。

### 3.2 `agentsdk/`

该目录放当前应用自带的 `agentsdk` 副本。

约束如下：

- `vulnhelper` 内部代码应优先引用这里的 SDK
- 不再直接依赖仓库根目录下的 `agentsdk`
- 若需要升级 SDK，需同步评估 vendored 副本与上游实现的差异
- 该副本只保留运行时必需文件，不保留上游 `tests/` 与 `__pycache__/`

### 3.3 `data/`

该目录只放静态数据源与后续离线数据文件。

- `vulns.db`
  当前唯一权威漏洞数据库。路径固定为 `agent_apps/vulnhelper/data/vulns.db`。

后续若加入同目录下的辅助映射表，应保持只读和可重建，不放运行时缓存。

### 3.4 `agents/`

该目录只放对 `agentsdk.Agent` 的封装，不放业务状态机。

- `planner.py`
  创建 PlannerAgent。
- `executor.py`
  创建 ExecutorAgent，并挂载工具与 hook。
- `analyst.py`
  创建 AnalystAgent。
- `factory.py`
  统一提供 `build_planner_agent()`、`build_executor_agent()`、`build_analyst_agent()`。

### 3.5 `application/`

该目录是业务编排核心。

- `orchestrator.py`
  负责四阶段主流程与状态机流转。
- `session_manager.py`
  负责业务会话对象创建、读取、更新。
- `request_router.py`
  根据用户输入判断是“新查询”“确认”“补充条件”还是“下钻过滤”。
- `dto.py`
  定义应用层输入输出 DTO，隔离内部领域对象与外部接口。

### 3.6 `domain/`

该目录放不依赖 `agentsdk` 的纯业务逻辑。

- `enums.py`
  定义会话状态、查询模式、用户目标、风险等级等枚举。
- `models.py`
  定义 `QueryPlan`、`CachedQueryResult`、`QuerySummary`、`FilterSpec`、`VulnRecord` 等领域对象。
- `normalization.py`
  做包名、CVE、风险等级和用户表达归一化。
- `versioning.py`
  负责版本解析、范围命中判断、固定版本比较。
- `record_parser.py`
  负责把 DB 原始行解析为领域对象，并从 `impact.vendors_products` 中提取范围信息。
- `filtering.py`
  做风险、POC、修复方案、恶意包等过滤。
- `summarizer.py`
  负责从匹配结果中汇总报告需要的核心指标。
- `confirmation.py`
  负责将 `QueryPlan` 变成面向用户的确认摘要数据。

### 3.7 `infra/`

该目录负责外部依赖与运行时基础设施。

- `repository.py`
  定义 Repository 抽象接口。
- `sqlite_repository.py`
  基于 SQLite 的具体实现。
- `session_cache.py`
  当前会话缓存实现，初版为内存级。
- `event_logger.py`
  订阅 `agentsdk` 事件并记录摘要日志。
- `clock.py`
  提供可注入时间源，方便测试缓存时间与过期逻辑。

### 3.8 `tools/`

该目录只放 `agentsdk` 工具对象及其 schema。

- `query_vulns.py`
  阶段二主工具。
- `filter_cached_results.py`
  阶段四过滤工具。
- `schemas.py`
  统一定义工具输入输出 schema，避免散落在多个文件。

### 3.9 `renderers/`

该目录负责最终输出的文本渲染。

- `confirmation_renderer.py`
  渲染阶段一确认话术。
- `report_renderer.py`
  渲染阶段三完整 Markdown 报告。
- `table_renderer.py`
  渲染阶段四紧凑表格。
- `markdown_helpers.py`
  公共表格、转义、标题生成辅助函数。

### 3.10 `prompts/`

该目录放系统提示词和输出协议说明。

- `planner_system.txt`
- `executor_system.txt`
- `analyst_system.txt`
- `output_contracts.md`

其中 `output_contracts.md` 用于描述各 Agent 输出的固定字段和格式约束，便于 prompt 共享。

### 3.11 `templates/`

该目录只放模板文件，不放业务代码。

- `report.md.j2`
- `confirmation.md.j2`
- `table.md.j2`

### 3.12 `tests/`

该目录同时包含样例数据与自动化测试。

- `vuln_agent_cases.json`
  输入问题样例。
- `vuln_golden_answers.json`
  期望输出基线。
- 其余 `test_*.py`
  覆盖领域逻辑、工具、编排与 golden 回归。

## 四、依赖方向

目录之间依赖必须遵循单向规则：

```text
app/bootstrap
  -> application
  -> agents
  -> agentsdk
  -> infra
  -> renderers

application
  -> domain
  -> infra
  -> agents
  -> agentsdk
  -> renderers

agents
  -> tools
  -> prompts
  -> domain
  -> agentsdk

tools
  -> domain
  -> infra

renderers
  -> domain
  -> templates

infra
  -> domain
```

不允许的依赖包括：

- `domain` 反向依赖 `application` 或 `agents`
- `tools` 直接依赖 `renderers`
- `renderers` 直接查询 SQLite

## 五、建议的最小落地顺序

若要按目录逐步实现，推荐顺序如下：

1. `domain/`
2. `infra/repository.py`、`infra/sqlite_repository.py`
3. `infra/session_cache.py`
4. `tools/`
5. `renderers/`
6. `agents/`
7. `application/`
8. `bootstrap.py`、`app.py`
9. `tests/`

这样可以先把确定性逻辑做稳，再接 `agentsdk` 编排层。
